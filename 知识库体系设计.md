# 数据治理平台知识库体系设计

> **文档定位**：定义支撑隐患识别报告输出的知识库结构，确保报告内容来自结构化知识库匹配，而非生成式AI自由生成。
> **设计原则**：从最终输出结果倒推知识库设计，确保每个输出字段都有明确的数据来源。
> **版本**：v1.0
> **更新日期**：2026-03-23

---

## 1. 设计背景与约束

### 1.1 产品约束
根据《数据治理平台产品上下文.md》：
- **AI职责边界**：AI负责隐患识别；法规依据与整改建议由结构化知识库匹配；生产报告文本不由生成式AI自由生成。

### 1.2 设计目标
- 建立结构化知识库体系，支撑隐患识别报告的自动化生成
- 确保报告内容的专业性、准确性和可追溯性
- 支持知识库的持续维护和迭代

## 2. 最终输出展示（终局视角）

### 2.1 隐患识别报告示例

```
【某工地隐患排查报告】

发现隐患：
1. 防护栏杆高度不足
   - 隐患描述：防护栏杆高度不足（低于1.2m）
   - 位置：[照片中的框选区域]
   - 隐患等级：重大
   - 法规依据：《建筑施工高处作业安全技术规范》JGJ80-2016 第4.1.2条
   - 整改建议：加高栏杆至1.2m以上，确保横杆符合要求
   - 整改期限：立即整改
   - 警示案例：2024年6月，某市某建筑工地因临边防护栏杆高度不足，
     一名工人从4楼坠落身亡。（来源：XX市应急管理局事故通报）

2. 洞口未设置防护
   - 隐患描述：洞口未设置盖板或防护栏杆
   - 位置：[照片中的框选区域]
   - 隐患等级：重大
   - 法规依据：《建筑施工高处作业安全技术规范》JGJ80-2016 第4.2.1条
   - 整改建议：根据洞口尺寸采取盖板或栏杆防护措施，盖板需固定牢固
   - 整改期限：立即整改
   - 警示案例：2023年11月，某工地楼板预留洞口未设置防护，
     施工人员夜间作业时坠入洞口致重伤。（来源：XX省住建厅安全简报）
```

### 2.2 输出字段与数据来源映射

| 输出字段 | 数据来源 | 说明 |
|---------|---------|------|
| 隐患名称 | 隐患库.隐患名称 | AI识别出隐患编码后匹配 |
| 隐患描述 | 隐患库.隐患描述 | 标准化描述，通用于所有该类隐患 |

| 警示案例 | 隐患案例库（通过隐患编码关联） | 真实发生的灾害事故，用于警示 |

## 4. 知识库详细设计

### 4.1 隐患库（3级结构）

**用途**：定义所有可识别的隐患类型及其标准化描述，采用3级分类结构，是知识库体系的核心。

**分级说明**：
- **一级：事故类型** - 按照事故分类标准划分的大类（如高坠、物体打击、机械伤害、触电、坍塌等）
- **二级：检查点位** - 具体的检查位置类型，便于现场巡查时快速定位（如临边洞口、脚手架、基坑边坡等）
- **三级：具体隐患** - 最细粒度的隐患类型，AI识别的目标（如防护栏杆高度不足、防护栏杆缺少踢脚板等）

**包含信息**：

| 字段名称 | 字段描述 | 示例内容 |
|---------|---------|---------|
| 隐患编码 | 唯一标识，包含3级层级信息 | FALL-EDGE-001 |
| 一级分类（事故类型） | 隐患所属的事故大类 | 高坠 |
| 二级分类（检查点位） | 隐患发生的具体位置类型 | 临边洞口 |
| 隐患名称 | 隐患的简短名称（三级） | 防护栏杆高度不足 |
| 隐患描述 | 标准化的隐患描述，用于报告输出 | 防护栏杆高度不足（低于1.2m） |
| 隐患等级 | 隐患的严重程度 | 重大、较大、一般 |
| 整改措施 | 标准化的整改操作步骤 | 加高栏杆至1.2m以上，确保横杆符合要求 |
| 整改期限 | 整改的紧急程度，固定选项 | 立即整改、限期整改 |
| 创建时间 | 该隐患类型首次录入的时间 | 2026-03-23 10:00:00 |
| 更新时间 | 最后一次修改的时间 | 2026-03-23 10:00:00 |
| 创建人 | 录入该隐患类型的人员 | 张三 |
| 更新人 | 最后修改的人员 | 张三 |
| 状态 | 该隐患类型是否在使用中 | 启用、停用 |
| 备注 | 其他需要说明的信息 | - |

**编码规则说明**：
```
FALL-EDGE-002
 │    │    └─ 三级：具体隐患序号
 │    └────── 二级：检查点位代码
 └─────────── 一级：事故类型代码

常用代码：
- 事故类型：FALL(高坠)、STRIKE(物体打击)、MECH(机械伤害)、ELEC(触电)、COLLAPSE(坍塌)
- 检查点位：EDGE(临边洞口)、SCAFFOLD(脚手架)、PIT(基坑边坡)、ELECBOX(配电箱)
```

**示例**：

```
隐患编码：FALL-EDGE-002
一级分类（事故类型）：高坠
二级分类（检查点位）：临边洞口
隐患名称：防护栏杆高度不足
隐患描述：防护栏杆高度不足（低于1.2m）
隐患等级：重大
整改措施：加高栏杆至1.2m以上，确保横杆符合要求
整改期限：立即整改
创建时间：2026-03-23 10:00:00
更新时间：2026-03-23 10:00:00
创建人：张三
更新人：张三
状态：启用
备注：-
```


### 4.3 隐患-法规映射表

**用途**：建立隐患与法规的关联关系，一个隐患可以对应多条法规。

**包含信息**：

| 字段名称 | 字段描述 | 示例内容 |
|---------|---------|---------|
| 映射ID | 系统内部唯一标识 | MAP-001 |
| 隐患编码 | 关联的隐患编码 | FALL-EDGE-001 |
| 条款ID | 关联的法规条款ID | STD-001 |
| 关联类型 | 该法规与隐患的关系类型 | 主要依据、参考依据、相关条款 |
| 创建时间 | 该映射关系首次录入的时间 | 2026-03-23 10:00:00 |
| 更新时间 | 最后一次修改的时间 | 2026-03-23 10:00:00 |
| 创建人 | 录入该映射关系的人员 | 张三 |
| 更新人 | 最后修改的人员 | 张三 |
| 备注 | 其他需要说明的信息 | - |

**关联类型判断标准**：

| 关联类型 | 判断标准 | 举例 |
|---------|---------|------|
| 主要依据 | 条款直接明确规定该隐患的具体技术要求（如尺寸、数量、操作规程），违反即构成该隐患 | JGJ80-2016 第4.1.2条："防护栏杆高度不得低于1.2m"（直接规定高度要求） |
| 参考依据 | 条款从更宏观角度涉及该隐患，或规定了相关管理要求，但不是直接的技术规定 | JGJ59-2011 安全检查标准中涉及临边防护的评分条款 |
| 相关条款 | 条款与该隐患有关联但并非直接依据，可用于扩展说明 | 相关施工组织设计或方案审批要求 |

> 关联类型由领域工程师或安全专家在录入知识库时判断，不由系统自动判断。

**报告展示规则**：

| 内容类型 | 展示数量 | 优先规则 |
|---------|---------|---------|
| 法规依据 | 最多2条 | 优先展示"主要依据"类型；如无主要依据则展示第一条"参考依据" |
| 警示案例 | 1条 | 优先选择事故后果为"死亡"的案例；多条时取创建时间最近的一条 |

**示例：防护栏杆高度不足的法规映射**

```
映射1：
  映射ID：MAP-001
  隐患编码：FALL-EDGE-002（防护栏杆高度不足）
  条款ID：STD-001（JGJ80-2016 第4.1.2条）
  关联类型：主要依据
  创建时间：2026-03-23 10:00:00
  更新时间：2026-03-23 10:00:00
  创建人：张三
  更新人：张三
  备注：-

映射2：
  映射ID：MAP-002
  隐患编码：FALL-EDGE-002（防护栏杆高度不足）
  条款ID：STD-002（建筑施工安全检查标准）
  关联类型：参考依据
  创建时间：2026-03-23 10:00:00

  创建人：张三
  更新人：张三
```

### 4.4 隐患案例库

**用途**：存储真实发生的灾害事故案例，与隐患类型关联，在报告中展示用于警示教育，让相关人员认识到隐患的严重后果。

**包含信息**：

| 字段名称 | 字段描述 | 示例内容 |
|---------|---------|---------|
| 案例ID | 系统内部唯一标识 | CASE-001 |
| 隐患编码 | 关联的主要隐患编码（一个案例对应一个主要隐患） | FALL-EDGE-002 |
| 事故标题 | 事故的简短标题 | 某市某工地因防护栏杆高度不足致工人坠亡 |
| 事故摘要 | 事故经过的简要描述，用于报告输出 | 2024年6月，某市某建筑工地因临边防护栏杆高度不足，一名工人从4楼坠落身亡 |
| 事故后果 | 事故造成的人员伤亡情况 | 1人死亡 |
| 来源链接 | 原始信息的网址（如有） | https://... |
| 填充方式 | 案例的录入来源 | AI搜索、人工录入 |
| 创建时间 | 案例录入的时间 | 2026-03-23 10:00:00 |
| 更新时间 | 最后一次修改的时间 | 2026-03-23 10:00:00 |
| 创建人 | 录入该案例的人员 | 张三 |
| 更新人 | 最后修改的人员 | 张三 |
| 状态 | 该案例是否在使用中 | 启用、停用 |
| 备注 | 其他需要说明的信息 | - |

**案例库填充机制**：

隐患案例库支持 AI 自动搜索填充，流程如下：

```
1. 触发搜索
   根据隐患编码对应的隐患名称（如"防护栏杆高度不足"），
   AI 自动在网络上搜索相关真实事故案例

2. 自动填写
   AI 按字段要求填写：
   - 事故标题、事故摘要、事故后果、来源链接
   - 填充方式标记为"AI搜索"
   - 状态默认为"停用"（待审核）

3. 人工复核
   领域工程师或安全专家对 AI 填充的内容进行审核：
   - 确认事故信息真实、来源链接有效
   - 审核通过后将状态改为"启用"
   - 信息有误则直接删除或修正
```

> AI 填充的案例必须经过人工复核后方可启用，未审核内容不会出现在报告中。

**示例**：

```
案例ID：CASE-001
隐患编码：FALL-EDGE-002
事故标题：某市某工地因防护栏杆高度不足致工人坠亡
事故摘要：2024年6月，某市某建筑工地因临边防护栏杆高度不足，一名工人从4楼坠落身亡
事故后果：1人死亡
```

## 5. 知识库维护机制

### 5.1 知识库更新场景

| 场景 | 涉及知识库 | 触发条件 |
|------|-----------|---------|---------|
| 新增隐患类型 | 隐患库、映射表 | 标注过程中发现未覆盖的隐患类型 |
| 法规更新 | 法律法规库、映射表 | 法规条款修订或新增 |
| 法规废止 | 法律法规库 | 法规被新版本替代 |
| 整改方案优化 | 隐患库 | 实践中发现更优的整改方案 |
| 隐患描述优化 | 隐患库 | 描述不够准确或专业 |
| 新增隐患案例 | 隐患案例库 | 收集到新的真实事故案例 |

### 5.2 知识库回流机制

根据业务流程（参考《数据治理平台-Mermaid.mmd》中的知识库回流子流程 P1-P5）：

```
P1: 发现未覆盖隐患（标注时识别）
    ↓
P2: 提报（标注员/质检员提交新隐患申请）
    ↓
P3: 审核（领域工程师审核）
    ↓
P4: 更新知识库（审核通过后更新）
    ↓
P5: 回写口径（更新标注/质检规范）
```

**实施要点**：
- 标注员在标注过程中发现无法归类的隐患，可提交"新隐患申请"
- 领域工程师审核后，在隐患库中新增记录
- 同步更新映射表
- 通知标注团队和质检团队更新标注口径

### 5.3 知识库版本管理

**版本控制策略**：
- 当前阶段以各库的"更新时间+更新人"字段作为最小可追溯信息
- 历史内容变更后无法自动还原，如需追溯请在变更前人工备份相关记录
- 知识库快照和报告与知识库版本绑定的能力规划在 v2.x 阶段实现

## 6. 关键设计原则

### 6.4 职责边界清晰

| 组件 | 职责 | 不负责 |
|------|------|--------|
| AI模型 | 识别照片中的隐患，输出隐患编码和位置 | 不生成文字描述、不判断法规依据、不提供整改建议 |
| 知识库 | 提供隐患的标准化描述、法规依据、整改建议、警示案例 | 不识别照片内容 |
| 报告生成模块 | 组装知识库匹配结果，生成结构化报告 | 不自由生成文本内容 |

### 6.2 数据标准化

- **隐患描述**：标准化、通用化，不涉及具体场景
- **法规依据**：完整引用法规名称、编号、条款编号
- **整改建议**：具体、可执行、可验收

### 6.1 可追溯性

- 报告中每个知识库匹配字段都有明确来源（见2.2输出字段映射表）
- 当前阶段以"更新时间+更新人"字段记录变更信息，支持基本审计
- 历史报告内容与当时知识库版本的精确绑定能力规划在 v2.x 阶段实现

### 6.3 可维护性

- 知识库结构清晰，易于理解和维护
- 支持增量更新，不影响已有数据
- 提供知识库回流机制，持续优化

## 7. 实施建议

### 7.1 初始化数据准备

**优先级 P0（v1.0 必须）**：
1. 梳理小散工程常见隐患类型（建议 20-30 种）
2. 收集相关法规条款（建议 50-100 条）
3. 编写标准化隐患描述和整改建议
4. 建立隐患-法规映射关系

**优先级 P1（v1.x 补充）**：
1. 收集真实灾害事故案例到隐患案例库
2. 优化隐患描述和整改建议

### 7.2 数据质量保障

- 隐患描述由领域工程师编写，确保专业性
- 法规条款由法务或合规人员审核，确保准确性
- 整改建议由安全专家审核，确保可执行性
- 定期评审知识库内容，及时更新
