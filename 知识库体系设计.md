# 数据治理平台知识库体系设计

> **文档定位**：定义支撑隐患识别报告输出的知识库结构，确保报告内容来自结构化知识库匹配，而非生成式AI自由生成。
> **设计原则**：从最终输出结果倒推知识库设计，确保每个输出字段都有明确的数据来源。
> **版本**：v1.0
> **更新日期**：2026-03-23

---

## 1. 设计背景与约束

### 1.1 产品约束
根据《数据治理平台产品上下文.md》：
- **AI职责边界**：AI负责隐患识别；法规依据与整改建议由结构化知识库匹配；生产报告文本不由生成式AI自由生成。

### 1.2 设计目标
- 建立结构化知识库体系，支撑隐患识别报告的自动化生成
- 确保报告内容的专业性、准确性和可追溯性
- 支持知识库的持续维护和迭代

## 2. 最终输出展示（终局视角）

### 2.1 隐患识别报告示例

```
【某工地隐患排查报告】

发现隐患：
1. 高处作业未系安全带
   - 隐患描述：作业人员在高处作业时，未按规定正确佩戴并系挂安全带，存在高处坠落风险
   - 位置：[照片中的框选区域]
   - 隐患等级：重大隐患
   - 法规依据：《建筑施工高处作业安全技术规范》JGJ80-2016 第3.0.2条
   - 整改建议：立即停工整改，作业人员必须系好安全带并挂在牢固的挂点上
   - 整改期限：立即整改

2. 临边防护缺失
   - 隐患描述：施工现场临边作业区域未设置防护栏杆或防护栏杆不符合规范要求，存在人员坠落风险
   - 位置：[照片中的框选区域]
   - 隐患等级：较大隐患
   - 法规依据：《建筑施工安全检查标准》JGJ59-2011 第4.2.1条
   - 整改建议：在临边设置1.2米高防护栏杆，设置中间横杆和挡脚板，确保防护栏杆牢固可靠
   - 整改期限：3日内完成
```

### 2.2 输出字段与数据来源映射

| 输出字段 | 数据来源 | 说明 |
|---------|---------|------|
| 隐患名称 | 隐患库.隐患名称 | AI识别出隐患ID后查询 |
| 隐患描述 | 隐患库.隐患描述 | 标准化描述，通用于所有该类隐患 |
| 隐患等级 | 隐患库.隐患等级 | 重大/较大/一般 |
| 法规依据 | 标准库（通过映射表关联） | 法规名称+编号+条款编号 |
| 整改建议 | 整改建议库.整改措施 | 标准化整改方案 |
| 整改期限 | 整改建议库.整改期限 | 立即/限期（天数） |

## 4. 知识库详细设计

### 4.1 隐患库

**用途**：定义所有可识别的隐患类型及其标准化描述，是知识库体系的核心。

**包含信息**：

| 字段名称 | 字段描述 | 示例内容 |
|---------|---------|---------|
| 隐患ID | 系统内部唯一标识 | HZD-001 |
| 隐患编码 | 便于管理的编码规则 | HZD-HIGH-001 |
| 事故类型 | 隐患所属的事故大类 | 高坠、物体打击、机械伤害、触电、坍塌 |
| 检查点位 | 隐患发生的具体位置类型 | 临边洞口、脚手架、基坑边坡、配电箱、起重设备 |
| 隐患名称 | 隐患的简短名称 | 防护栏杆高度不足 |
| 隐患描述 | 标准化的隐患描述，用于报告输出 | 防护栏杆高度不足（低于1.2m） |
| 隐患等级 | 隐患的严重程度 | 重大、较大、一般 |
| 典型特征 | 帮助标注员识别该隐患的视觉特征 | 临边防护栏杆存在但高度明显低于1.2米标准 |
| 标注示例图片 | 典型案例的图片路径，用于培训和参考 | /examples/HZD-002.jpg |
| 创建时间 | 该隐患类型首次录入的时间 | 2026-03-23 10:00:00 |
| 更新时间 | 最后一次修改的时间 | 2026-03-23 10:00:00 |
| 创建人 | 录入该隐患类型的人员 | 张三 |
| 更新人 | 最后修改的人员 | 张三 |
| 状态 | 该隐患类型是否在使用中 | 启用、停用 |
| 备注 | 其他需要说明的信息 | - |

**示例**：

```
隐患ID：HZD-002
隐患编码：HZD-HIGH-002
事故类型：高坠
检查点位：临边洞口
隐患名称：防护栏杆高度不足
隐患描述：防护栏杆高度不足（低于1.2m）
隐患等级：重大
典型特征：临边防护栏杆存在但高度明显低于1.2米标准
标注示例图片：/examples/HZD-002.jpg
创建时间：2026-03-23 10:00:00
更新时间：2026-03-23 10:00:00
创建人：张三
更新人：张三
状态：启用
```

### 4.2 标准库（法规条款库）

**用途**：存储法规条款，为隐患识别提供法规依据。

**包含信息**：

| 字段名称 | 字段描述 | 示例内容 |
|---------|---------|---------|
| 条款ID | 系统内部唯一标识 | STD-001 |
| 法规名称 | 法规的完整名称 | 建筑施工高处作业安全技术规范 |
| 法规编号 | 法规的官方编号 | JGJ80-2016 |
| 条款编号 | 具体条款的编号 | 第3.0.2条 |
| 条款内容 | 条款的完整文字内容 | 高处作业人员必须按规定正确佩戴安全带，安全带应高挂低用，挂点应牢固可靠 |
| 适用场景说明 | 该条款适用的具体场景 | 所有2米以上高处作业场景 |
| 法规类型 | 法规的级别分类 | 国家标准、行业标准、地方标准、企业标准 |
| 生效日期 | 该法规开始生效的日期 | 2016-12-01 |
| 废止日期 | 该法规被废止的日期（如已废止） | - |
| 状态 | 该法规是否仍在使用 | 现行、已废止 |
| 备注 | 其他需要说明的信息 | - |

**示例：JGJ80-2016 第3.0.2条**

```
条款ID：STD-001
法规名称：建筑施工高处作业安全技术规范

状态：现行
```

| 优先级 | 在报告中显示的优先顺序（数字越小优先级越高） | 1 |

### 5.2 SQL查询示例

#### 步骤1：查询隐患基本信息

```sql
SELECT
    hazard_name,
    hazard_description,
    hazard_level
FROM hazard_library
WHERE hazard_id = 'HZD-001' AND status = '启用';
```

#### 步骤2：查询法规依据

```sql
SELECT
    s.regulation_name,
    s.regulation_code,
    s.clause_number,
    s.clause_content,
    m.relation_type,
    m.priority
FROM hazard_regulation_mapping m
JOIN standard_library s ON m.clause_id = s.clause_id
WHERE m.hazard_id = 'HZD-001' AND s.status = '现行'
ORDER BY m.priority ASC;
```

SELECT

    deadline_days,
    acceptance_criteria,

WHERE hazard_id = 'HZD-001';

## 6. 知识库维护机制

### 6.1 知识库更新场景

| 场景 | 涉及知识库 | 操作类型 | 触发条件 |
|------|-----------|---------|---------|
| 新增隐患类型 | 隐患库、映射表、整改建议库 | INSERT | 标注过程中发现未覆盖的隐患类型 |
| 法规更新 | 标准库、映射表 | UPDATE/INSERT | 法规条款修订或新增 |
| 法规废止 | 标准库 | UPDATE | 法规被新版本替代 |
| 整改方案优化 | 整改建议库 | UPDATE | 实践中发现更优的整改方案 |
| 隐患描述优化 | 隐患库 | UPDATE | 描述不够准确或专业 |
| 案例积累 | 隐患案例库 | INSERT | 发现典型案例或困难样本 |

### 6.2 知识库回流机制

根据业务流程（参考《数据治理平台-Mermaid.mmd》中的知识库回流子流程 P1-P5）：

```
P1: 发现未覆盖隐患（标注时识别）
    ↓
P2: 提报（标注员/质检员提交新隐患申请）
    ↓
P3: 审核（领域工程师审核）
    ↓
P4: 更新知识库（审核通过后更新）
    ↓
P5: 回写口径（更新标注/质检规范）
```

**实施要点**：
- 标注员在标注过程中发现无法归类的隐患，可提交"新隐患申请"
- 领域工程师审核后，在隐患库中新增记录
- 同步更新映射表和整改建议库
- 通知标注团队和质检团队更新标注口径

### 6.3 知识库版本管理

**版本控制策略**：
- 每次知识库更新记录版本号（如 v1.0, v1.1, v1.2）
- 保留历史版本数据，支持追溯
- 导出训练数据时记录使用的知识库版本

**版本记录表**：

```sql
CREATE TABLE knowledge_base_version (
    version_id VARCHAR(50) PRIMARY KEY,
    version_number VARCHAR(20) NOT NULL,
    update_type ENUM('新增隐患', '法规更新', '整改优化', '描述修正'),
    update_description TEXT,
    affected_tables VARCHAR(200),
    updated_by VARCHAR(100),
    updated_at DATETIME
);
```

## 7. 关键设计原则

### 7.1 职责边界清晰

| 组件 | 职责 | 不负责 |
|------|------|--------|
| AI模型 | 识别照片中的隐患，输出隐患ID和位置 | 不生成文字描述、不判断法规依据、不提供整改建议 |
| 知识库 | 提供隐患的标准化描述、法规依据、整改建议 | 不识别照片内容 |
| 报告生成模块 | 组装知识库查询结果，生成结构化报告 | 不自由生成文本内容 |

### 7.2 数据标准化

- **隐患描述**：标准化、通用化，不涉及具体场景
- **法规依据**：完整引用法规名称、编号、条款编号
- **整改建议**：具体、可执行、可验收

### 7.3 可追溯性

- 每条报告内容都能追溯到知识库的具体记录
- 知识库更新有版本记录和审批流程
- 支持审计和质量回溯

### 7.4 可维护性

- 知识库结构清晰，易于理解和维护
- 支持增量更新，不影响已有数据
- 提供知识库回流机制，持续优化

## 8. 实施建议

### 8.1 初始化数据准备

**优先级 P0（v1.0 必须）**：
1. 梳理小散工程常见隐患类型（建议 20-30 种）
2. 收集相关法规条款（建议 50-100 条）
3. 编写标准化隐患描述和整改建议
4. 建立隐患-法规映射关系

**优先级 P1（v1.x 补充）**：
1. 积累典型案例到案例库
2. 补充困难样本
3. 优化隐患描述和整改建议

### 8.2 数据质量保障

- 隐患描述由领域工程师编写，确保专业性
- 法规条款由法务或合规人员审核，确保准确性
- 整改建议由安全专家审核，确保可执行性
- 定期评审知识库内容，及时更新

### 8.3 技术实现建议

- 使用关系型数据库（MySQL/PostgreSQL）存储知识库
- 为高频查询字段建立索引（hazard_id, clause_id）
- 实现知识库缓存机制，提升查询性能
- 提供知识库管理后台，支持可视化维护
