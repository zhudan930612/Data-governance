# 变更上下文文档

## 技能作用

- 作为上下文主文档的默认维护入口，保障业务背景信息长期可持续更新。
- 将新增信息按证据增量合并到既有基线，避免每次重写导致口径漂移。
- 通过标准化校验与更新记录，确保每次变更可追溯、可复盘、可同步到下游。

## 使用场景

- 已有 `产品上下文.md`，需要根据新会议结论、调研结果或业务变化做增量修订。
- 术语口径、核心痛点、用户职责或商业模式出现变化，需要更新并统一表述。
- 需要修正旧版本中的错误或过期事实，同时保留变更依据与废弃说明。
- 一次更新后需要明确下游影响（流程/架构/功能）并给出同步建议。
- 需要对现有上下文文档做口径一致性审查，并判断是否可冻结。

## 入口对齐规则

- 已有上下文主文档时，`update-context-doc` 为默认入口。
- `generate-context-doc` 仅用于首次生成；除非用户明确要求“重建/全量重写”，否则不应替代本技能。


## 触发词与模式映射

- 更新/增量更新/同步：`mode=更新`
- 调整/修订/优化表述：`mode=调整`
- 审查/复核/校验/一致性检查：`mode=审查`

> 三种模式共用同一套输入边界、门禁规则与一致性校验。

按以下步骤执行上下文变更或审查。

## 0. 通用规则（内置）

- 仅使用可见来源：现有文档、用户描述、截图、会议纪要。
- 禁止补充未被证实的事实；信息缺失标注“待补充”。
- 核心痛点必须包含 4 要素：业务场景对象、问题现象、业务影响、来源依据。
- 关键术语必须包含 3 要素：定义、边界（包含/不包含）、来源依据。
- 术语冲突统一标记：`[口径冲突: 术语X与痛点Y中的表述不一致]`。
- 正文标题不使用括号；正文主标题下必须有可点击目录；目录仅包含实际生成章节。
- “更新记录”为正文必填且位于文末；字段格式以第 5 节为唯一准。
- 来源与输入不单列章节，统一写入“更新记录”。
- “非目标”不在正文中作为独立标题展示，仅作为生成约束。
- 禁止写入实现层细节与时间敏感高频变化信息。

## 1. 锁定变更输入

- 收集本次变更来源（会议结论、截图、现有文档、用户补充说明）。
- 标注来源粒度（文件名/章节/时间点）。
- 无来源证据的变更请求，标注“待补充”并暂不落盘为事实。

### 1.1 变更类型判定（先分类再更新）

- 事实新增：来源确认的新业务事实进入文档。
- 事实修正：来源显示旧事实错误或过期，需替换修正。
- 口径澄清：定义更清晰但结论不变，更新术语或描述边界。
- 口径废弃：旧口径不再适用，需废弃并给出替代口径。
- 结构优化：不改事实，仅优化章节结构与表达可读性。

### 1.2 执行前门禁（不通过不落盘）

- 是否有明确来源证据。
- 是否属于上下文层（而非流程/架构/功能实现层）。
- 是否影响核心痛点或关键术语口径。
- 任一条件无法判断时，不落盘事实，输出“待确认项”清单。

## 2. 判断是否属于上下文层

仅当变更属于以下类型时进入上下文：

- 业务背景变化
- 商业模式变化
- 核心用户职责变化
- 术语口径变化
- 全局原则变化
- 当前用户与业务现状问题变化

以下内容不应写入上下文（转交对应层）：

- 业务流程细节（转流程文档）
- 模块能力/优先级细节（转架构规划文档）
- 页面与功能实现细节（转功能设计文档）

## 3. 最小化修改策略

- 仅改受影响段落，不做无关重写。
- 保持原文风格和稳定结构。
- 新增内容必须能被来源直接支撑。
- 删除内容要有明确原因（口径废弃、事实变化、重复冲突）。

### 3.1 段落级合并动作（增量更新）

- `Add`：新增段落，仅补充当前缺失且有来源支撑的信息。
- `Edit`：编辑已有段落，仅修改受影响句段，不扩散改动范围。
- `Deprecate`：废弃旧口径，不直接静默删除；需在“更新记录”写明废弃原因与替代口径。

## 4. 强制一致性校验

- 新旧口径是否冲突。
- 若存在口径冲突，是否使用统一标记 `[口径冲突: 术语X与痛点Y中的表述不一致]`。
- 问题与解决思路是否仍是业务场景层表达。
- 痛点相关改动是否仍满足“场景对象 + 问题现象 + 业务影响 + 来源依据”。
- 术语相关改动是否仍满足“定义 + 边界 + 来源依据”。
- 是否混入实现层内容。
- 是否出现无来源新增事实。
- 是否存在未确认信息且未标注“待补充”。
- 核心痛点、术语与产品价值主张是否保持一致。
- 若标题发生变更，目录项与锚点链接是否已同步更新。

## 5. 变更输出

- `mode=审查` 且无明确“直接改文档”指令时，默认不落盘正文，仅输出审查结论与修复建议。

输出时固定包含以下四段（闭环输出）：

- 变更摘要：改了哪些段落、采用了哪些动作（`Add/Edit/Deprecate`）。
- 证据摘要：依据哪些来源，分别支撑哪些改动。
- 风险与待确认：哪些内容仍待补充，哪些判断尚未满足门禁条件。
- 下游建议：是否影响流程/架构/功能文档，以及建议同步项。

补充要求：

- 来源与输入信息只写入“更新记录”，不新增“来源说明/输入说明”独立章节。
- “更新记录”为正文必填章节，固定在文档末尾。
- 建议格式：`版本 | 日期 | 变更类型 | 影响章节 | 来源摘要 | 下游影响 | 更新内容`。
- “更新内容”需包含：改动范围摘要 + 关键决策 + 必要的废弃说明（如有）。


### 5.1 审查模式必填输出块（对话输出）

- 上下文口径一致性：通过/不通过
- 证据可追溯性：通过/不通过
- 跨层边界合规性：通过/不通过
- 阻塞项与修复建议

> 判定词必须使用“通过 / 不通过”，禁止使用模糊表述。

## 6. 测试场景

- 审查模式：在不落盘正文时，仍可输出完整“通过/不通过 + 阻塞项 + 修复建议 + 待确认项”。
- 无来源变更请求：标注“待补充”并阻断落盘。
- 跨层污染场景：发现流程/架构/实现细节混入时，输出不通过并给出分流建议。
- 口径冲突场景：按统一冲突标记输出，并给出修复路径。

## 禁止项

- 禁止“为了完整”补充无证据内容。
- 禁止把个人理解替代来源事实。
- 禁止将技术实现细节写入上下文。
- 禁止跨层覆盖下游文档职责边界。
- 禁止绕过本技能规则自行新增口径。
