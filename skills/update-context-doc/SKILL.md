# 变更上下文文档

## 入口对齐规则

- 已有上下文主文档时，`update-context-doc` 为默认入口。
- `generate-context-doc` 仅用于首次生成；除非用户明确要求“重建/全量重写”，否则不应替代本技能。

按以下步骤执行上下文变更。

## 1. 锁定变更输入

- 收集本次变更来源（会议结论、截图、现有文档、用户补充说明）。
- 标注来源粒度（文件名/章节/时间点）。
- 无来源证据的变更请求，标注“待补充”并暂不落盘为事实。

## 2. 判断是否属于上下文层

仅当变更属于以下类型时进入上下文：

- 业务背景变化
- 商业模式变化
- 核心用户职责变化
- 术语口径变化
- 全局原则变化
- 当前用户与业务现状问题变化

以下内容不应写入上下文（转交对应层）：

- 业务流程细节（转流程文档）
- 模块能力/优先级细节（转架构规划文档）
- 页面与功能实现细节（转功能设计文档）

## 3. 最小化修改策略

- 仅改受影响段落，不做无关重写。
- 保持原文风格和稳定结构。
- 新增内容必须能被来源直接支撑。
- 删除内容要有明确原因（口径废弃、事实变化、重复冲突）。

## 4. 一致性校验（必做）

- 新旧口径是否冲突。
- 问题与解决思路是否仍是业务场景层表达。
- 是否混入实现层内容。
- 是否出现无来源新增事实。

## 5. 变更输出

输出时提供简短变更说明：

- 改了哪些段落
- 依据哪些来源
- 哪些内容待补充
- 是否影响下游文档（流程/架构/功能）

## 禁止项

- 禁止“为了完整”补充无证据内容。
- 禁止把个人理解替代来源事实。
- 禁止将技术实现细节写入上下文。
- 禁止跨层覆盖下游文档职责边界。
