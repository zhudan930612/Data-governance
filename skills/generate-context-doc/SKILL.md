# 生成上下文文档

## 文档作用

- 形成稳定、可复用的业务背景基线，作为后续流程设计、架构规划、功能设计的共同前提。
- 统一团队与 AI 对产品背景的认知口径，减少因背景理解偏差导致的返工。
- 明确“问题是什么、价值方向是什么”，为后续方案讨论提供边界与判断依据。

## 使用场景

- 新产品或新业务线启动时，需要快速沉淀一版可信的上下文基线文档。
- 现有背景信息分散在会议纪要、旧文档、截图中，需要归并成统一上下文。
- 进入流程梳理、架构规划、功能设计前，需要先校准业务背景与核心假设。
- 出现认知分歧时，需要基于已确认来源重建共同语义和业务边界。

## 语义触发说明

- 触发不要求逐字命中，按用户意图判定。
- 只要用户意图是“生成首版业务背景基线文档”，即可触发本技能。

按以下步骤生成上下文文档。

## 1. 明确输入边界

- 只使用可见来源：现有文档、用户描述、截图、会议纪要。
- 禁止补充未被证实的事实。
- 发现信息缺失时标注“待补充”，不要猜测。

## 2. 先确定产品名与目标路径

- 先确认产品名；若无法从输入材料明确获取，必须先向用户索取，不允许自拟。
- 固定主文档目标路径：仓库根目录 `[产品名]产品上下文.md`。
- 仓库根目录指当前 Git 仓库根目录，不是任意子目录或当前打开文件目录。

## 3. 执行前判断（首次生成 vs 增量维护）

- 基于第 2 步得到的目标路径检查文件是否存在。
- 若目标主文档已存在，默认不重写，提示改走 `update-context-doc` 做增量维护。
- 若 `update-context-doc` 不可用，需明确告知用户，并在当前会话按增量原则更新原文，不做全量重写。
- 仅当用户明确要求“重建/全量重写”时，才允许本技能全量输出并覆盖。

## 4. 提取可复用业务基线

从来源中提取并归并以下信息：核心章节必须覆盖，增强与可选章节按实际情况增减。

### 4.1 核心必填章节（必须覆盖）

- 业务背景与现状问题
- 产品定位与商业模式
- 核心用户与职责边界
- 关键术语及口径定义
- 全局设计原则与稳定约束
- 产品价值主张与业务解决路径

### 4.1.1 现状问题与核心痛点写作硬规则

- 每条核心痛点必须包含 4 个要素：业务场景对象、问题现象、业务影响、来源依据。
- 禁止只写抽象结论（如“效率低”“体验差”）而不说明具体场景和影响。
- 痛点必须是用户与业务现状问题，不得替换为模块实现缺口或技术债。
- 若来源不足以支撑完整要素，缺失部分必须标注“待补充”。

### 4.1.2 关键术语与口径定义硬规则

- 仅收录易混淆、跨角色高频使用、对决策有影响的术语。
- 每个术语必须包含 3 个要素：定义、边界（包含/不包含）、来源依据。
- 禁止循环定义、同义词互指、模糊词定义（如“一般”“大概”“通常”）。
- 术语口径必须与核心痛点、价值主张保持一致，冲突项必须显式标注。
- 冲突项统一使用标记格式：[口径冲突: 术语X与痛点Y中的表述不一致]。

### 4.2 增强建议章节（有信息就补）

> 硬规则：不纳入与时间强绑定、易频繁变化的内容（如当前业务阶段、口径变更记录、阶段性里程碑状态）。

- 业务范围
- 关键业务场景与使用边界
- 上下游关系与生态位置（对谁输入/对谁输出）
- 核心价值链路（问题 -> 方案 -> 价值）
- 成功判定口径（业务层，不写技术阈值）
- 关键假设与前提条件

### 4.3 可选章节（视项目复杂度）

- 外部约束摘要（政策/合同/行业/客户要求）
- 业务风险与依赖（业务层）
- 历史关键决策背景（为什么这么定）

## 5. 生成文档正文

- 先写“稳定信息”，再写“问题与思路”。
- 语言保持产品经理视角，面向开发可理解、可执行。
- 避免写实现细节（页面字段、接口参数、算法阈值、数据库结构）。
- 不要求固定模板，但必须覆盖对后续流程/架构/功能设计有指导价值的内容。
- 不在正文中输出“写作规则/校验规则/生成方法”等元说明，这类信息仅在对话核验摘要中表达。
- 正文标题不使用括号；括号中的限定语改写为正文描述或列表项。
- 不在正文中单独设置“来源说明/输入说明”章节；来源与输入仅记录在“更新记录”。
- “非目标”仅用于生成约束，不在正文中作为独立章节或小节展示。
- “非目标”相关信息可作为边界描述融入“文档定位与边界”或“业务范围”等正文段落，但不以“非目标”标题单列。
- 正文主标题下必须生成“目录”区块，使用可点击锚点链接到各二级标题。
- 若标题有增删改，目录项与锚点链接必须同步更新，确保可跳转。
- 目录只包含实际生成的章节；可选章节仅在实际生成时纳入目录。


### 5.1 更新记录规范

- “更新记录”为正文必填章节，位置固定在文档末尾。
- 建议章节名：`## 更新记录`。
- 建议格式使用表格：`版本 | 日期 | 更新内容`。
- 更新内容至少包含：本次改动范围摘要 + 来源/输入摘要。
- 不单独新增“来源说明/输入说明”章节，相关信息只写入“更新记录”。

## 6. 来源一致性自检（必做）

逐条检查：

- 是否存在来源中没有的新增事实。
- “现状问题/核心痛点”是否是用户与业务场景问题，而非模块实现缺口。
- 每条核心痛点是否具备“场景对象 + 问题现象 + 业务影响 + 来源依据”。
- 每个关键术语是否具备“定义 + 边界 + 来源依据”，且无循环定义或模糊定义。
- 术语口径是否与核心痛点、价值主张、业务边界一致。
- 若存在口径冲突，是否使用统一标记 [口径冲突: 术语X与痛点Y中的表述不一致]。
- 是否混入流程、架构、页面实现细节。
- 是否存在未确认信息且未标注“待补充”。
- 正文是否包含目录，且目录项与二级标题一一对应并可点击跳转。
- 目录是否仅包含实际生成章节，未生成的可选章节未被错误写入目录。
- 是否已在对话核验摘要中汇总全部“待补充”项及数量。

## 7. 输出要求

- 输出文档后，需附简短核验摘要（新增事实判断、是否全量重写、待补充项汇总）。
- 来源与输入信息不在正文单列，统一写入文档“更新记录”。
- 核验摘要只在对话消息中输出，不写入上下文文档本体。
- 核验摘要建议包含：核心痛点条目数、术语条目数、待补充项数量与清单。

## 禁止项

- 禁止把推断写成事实。
- 禁止把“可能、应该、大概率”表述伪装为确定结论。
- 禁止把模块缺陷当作“业务问题”写入上下文。
- 禁止复制流程/架构/功能文档的实现细节进上下文。
- 禁止写入时间敏感且高频变化的信息（如当前业务阶段、阶段目标、口径变更记录、临时状态结论）。
