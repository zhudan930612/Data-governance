## 1. 文档定位与分工

- 文档目标：在业务流程基础上，沉淀平台模块边界、主要子功能点、当前建设状态与后续 PRD 拆分入口，作为页面设计与后续研发输入前的中间层。
- 文档边界：仅描述产品架构层信息，不展开页面交互、字段结构、接口参数、数据库结构与研发实施方案。

## 2. 架构拆分原则

### 2.1 拆分依据

- 一级模块边界以 `数据治理平台业务流程总览表.md` 第 2.7 节“架构拆分输入清单”为准。
- 子功能点统一按“任务/页面型”拆解，围绕角色、工作台、处理对象、状态流转展开，而不是仅按流程步骤或技术能力命名。
- 所有流程引用统一使用 S/G/R/P 编号，不使用 Mermaid 变量名。

### 2.2 状态口径

- `待确认`：现有文档无法判断真实建设现状。
- `待规划`：已确认需要建设，但未见具体方案或落地证据。
- `设计中`：已有方案设计，但未见落地证据。
- `开发中`：已确认进入开发或实施阶段。
- `已上线`：已确认投入实际使用。
- `待优化`：已具备基础能力，但存在明确优化方向。
- `暂不建设`：当前阶段不纳入建设范围，仅保留流程占位。

> 根据当前已确认口径，整个平台仍处于规划阶段；如未出现更细粒度证据，模块与子功能状态默认使用 `待规划`。

### 2.3 编写约束

- 本文只记录可从现有文档推导出的模块目标、子功能点候选与状态证据。
- 不能因为某专题设计文档已存在，就直接推断平台中对应功能已上线。
- 模块状态与子功能状态在当前阶段统一以 `待规划` 为默认值；若后续出现更细粒度证据，再分别调整为 `设计中`、`开发中`、`已上线` 或 `待优化`。

## 3. 模块总览

### 3.1 模块层级关系图

```text
+----------------------+      +----------------------+
|   现场巡检平台       |----->|     数据接入管理     |
|      移动端          |      |     PC管理后台       |
|         S1           |      |       S2 / S3        |
+----------------------+      +----------------------+
                                      |
                                      v
+----------------------+       +----------------------+
|      生产作业层      |<----->|      规则支撑层      |
|    标注与质检协同    |       |   知识库与规则治理   |
|     PC管理后台       |       |     PC管理后台       |
|  S4 / S5 / G1 / R1   |       | P1 / P2 / P3 / P4 / |
|                      |       |        P5            |
+----------------------+       +----------------------+
           |                           |
           v                           v
+----------------------+<--------------+
|      治理运营层      |
|     数据治理运营     |
|     PC管理后台       |
|  S6 / S7 / G2 / R2   |
+----------------------+
           |
           v
+----------------------+
|      消费输出层      |
|    AI 训练与部署     |
|         S8           |
+----------------------+
```

- 主链路：现场巡检平台（移动端） -> 数据接入管理（PC管理后台） -> 标注与质检协同（PC管理后台） -> 数据治理运营（PC管理后台） -> AI 训练与部署。
- 侧向支撑：知识库与规则治理与标注、质检、数据治理运营双向联动。
- 双向关系含义：
  - `知识库 -> 数据治理运营`：知识库为 `G2/R2` 和强制复判提供判断依据与口径范围。
  - `数据治理运营 -> 知识库`：运营侧发现规则漂移、系统性问题或复查异常时，可反向触发 `P1-P5`。
- 终端边界：现场巡检平台是独立移动端模块；除现场巡检平台外，其余模块均为 PC 管理后台端。
- 使用方式：该图只表达模块层级示意；详细业务流转仍以 `数据治理平台-Mermaid.mmd` 和 `数据治理平台业务流程总览表.md` 为准。

## 4. 各模块说明

### 4.1 现场巡检平台

#### 4.1.1 模块定位

- 对应流程：S1。
- 模块职责：作为后续移动端来源入口，承接巡检员现场巡查、拍照、语音备注，并在后续阶段把现场图片同步到数据治理平台。
- 主要角色：巡检员。

#### 4.1.2 主要子功能点

1. `现场巡查记录`
   说明：承接巡检员在移动端的现场巡查记录与提交，作为后续阶段能力保留。
   当前建设状态：`暂不建设`
2. `现场拍照`
   说明：承接巡检员在现场拍摄隐患相关图片，作为后续阶段能力保留。
   当前建设状态：`暂不建设`
3. `语音备注`
   说明：承接巡检员在现场补充语音备注信息；首期不进入数据治理链路，仅作为后续来源入口能力保留。
   当前建设状态：`暂不建设`
4. `图片同步`
   说明：后续阶段将移动端采集的图片同步到数据治理平台；首期不建设弱网、离线缓存和自动补同步能力。
   当前建设状态：`暂不建设`

#### 4.1.3 当前判断

- 现场巡检平台长期保留为独立移动端模块，不并入 PC 管理后台。
- 首期不建设正式移动端链路，现场照片先按“手机拍照 -> 事后导出到 PC -> 数据接入管理导入”处理。
- 语音备注和图片同步作为后续来源入口能力保留，当前阶段不进入数据治理主链路。

#### 4.1.4 后续 PRD 拆分建议

- 后续阶段再编写 `PRD-现场巡检平台-现场巡查与拍照.md`
- 后续阶段再编写 `PRD-现场巡检平台-图片同步.md`

### 4.2 数据接入管理

#### 4.2.1 模块定位

- 对应流程：S2、S3。
- 模块职责：承接政府端/外部补充来源接入、现场照片 PC 导入、上传汇聚与样本池管理。
- 主要角色：政府端/外部数据提供方、数据管理员。
- 系统自动行为：导入接收、样本汇聚、来源标记写入。

#### 4.2.2 主要子功能点

1. `外部来源接入`
   说明：承接政府端/外部补充来源的数据进入平台。
   当前建设状态：`待规划`
2. `现场照片 PC 导入`
   说明：承接现场手机拍照后导出到 PC 的图片导入平台。
   当前建设状态：`待规划`
3. `上传汇聚`
   说明：将不同来源的图片统一汇聚到平台处理链路。
   当前建设状态：`待规划`
4. `样本池管理`
   说明：形成进入标注前的可处理样本池，并按导入任务维度管理批次边界。
   当前建设状态：`待规划`
5. `来源标记管理`
   说明：在接入阶段写入导入来源、导入批次、导入方式、来源责任人等基础来源锚点，不负责跨模块追溯查询展示。
   当前建设状态：`待规划`

#### 4.2.3 当前判断

- 该模块属于 PC 管理后台模块。
- 首期支持政府端/外部图片导入，以及现场手机拍照后导出到 PC 的图片导入。
- 现场巡检平台图片同步接收保留为后续能力，当前阶段不展开弱网/离线方案设计。
- 批次由每次导入任务自动生成，批次边界以单次导入任务为准，不按时间窗或人工重划。
- 单个样本只属于一个批次，不跨批次自动合并。
- 样本进入后统一进入样本池，样本池至少管理来源、上传时间、所属批次、状态、是否分配、是否标注、是否质检、是否入库。
- 来源标记管理只负责写入来源锚点；跨 S1/S2/S3/S4/S5/S7 的追溯查询与展示由横向基础能力中的全链路样本追溯承接。

#### 4.2.4 后续 PRD 拆分建议

- `PRD-数据接入管理-外部来源接入与PC导入.md`
- `PRD-数据接入管理-样本池与样本管理.md`

### 4.3 标注与质检协同

#### 4.3.1 模块定位

- 对应流程：S4、S5、G1、R1。
- 模块职责：承接多人多轮标注、全量质检裁决，以及不通过样本回流重标的闭环。
- 主要角色：数据标注团队、质检员、领域工程师（安全专家）、数据管理员。
- 系统自动行为：标注轮次分配、状态流转、挂起超时升级。

#### 4.3.2 主要子功能点

1. `标注池与轮次分配`
   说明：样本自动进入标注池，由系统分发到后续标注轮次，不做人工派发。
   当前建设状态：`待规划`
2. `标注工作台`
   说明：承接多人多轮标注，以及有隐患/无隐患两类处理。
   当前建设状态：`待规划`
3. `负样本确认`
   说明：只有满足最少配置轮次且各轮结论都为“无隐患”时，才进入负样本确认；任一轮出现“有隐患”则转入冲突裁决，不走负样本直通。
   当前建设状态：`待规划`
4. `质检裁决`
   说明：对标注结果进行全量质检，并承接多轮结果冲突时的冲突裁决，形成通过、驳回回流或挂起待确认三类结果。
   当前建设状态：`待规划`
5. `驳回回流`
   说明：将质检不通过样本按 R1 回流到 S4 重新标注。
   当前建设状态：`待规划`

#### 4.3.3 当前判断

- 已确认该模块必须形成“标注 -> 质检 -> 驳回回流 -> 再标注”的闭环。
- 样本不做人工派发，自动进入标注池。
- 标注轮次支持自定义配置“至少几轮”，并由系统自动分配给不同人员执行。
- 负样本确认只在达到最少配置轮次且各轮都判为“无隐患”时触发；任一轮出现“有隐患”则转入正常冲突裁决。
- 冲突裁决归入质检裁决，由质检工作台统一处理多轮标注结果冲突，不单列独立功能点或独立 PRD。
- 质检裁决首期保留“通过、驳回回流、挂起待确认”三种结果。
- 挂起待确认由质检员发起，领域工程师最终完成待确认判定；如因规则缺失无法裁决，则转入 P1-P5 知识库链路；如判断为非规则缺失，则直接恢复到 G1 继续裁决，不重走整轮质检处理。
- 挂起状态不算完成态，不允许作为批次长期滞留状态；首期必须具备超时升级机制，但暂不细化具体时长。
- 挂起超时后自动升级到领域工程师（安全专家）；只有其判断为规则缺失时，才再转入 P1-P5。
- 标注工作台与质检工作台拆成两套独立页面。

#### 4.3.4 后续 PRD 拆分建议

- `PRD-标注与质检协同-标注池与轮次分配.md`
- `PRD-标注与质检协同-标注工作台.md`
- `PRD-标注与质检协同-质检裁决与回流.md`

### 4.4 数据治理运营

#### 4.4.1 模块定位

- 对应流程：S6、S7、G2、R2。
- 模块职责：作为 PC 管理后台模块，承接高质量数据沉淀、入库、导出、抽查及问题回流，形成治理运营闭环。
- 主要角色：数据治理团队、质检员、领域工程师（安全专家）、AI 工程师、数据管理员。
- 系统自动行为：抽查周期触发、入池动作、留痕记录。

#### 4.4.2 主要子功能点

1. `高质量数据池`
   说明：G1 通过后的样本自动进入高质量数据池，作为治理与训练候选基础。
   当前建设状态：`待规划`
2. `入库与导出`
   说明：首期“入库”按进入高质量数据池表达；“导出”由数据治理侧按需手动发起，输出给 AI 工程师或后续训练侧使用。
   当前建设状态：`待规划`
3. `抽查与复查任务管理`
   说明：承接周/月抽查的发起、执行与结论记录，并承接由 `P5` 触发的强制复判复查任务生成与执行记录。
   当前建设状态：`待规划`
4. `问题回流`
   说明：将抽查发现的问题样本按 R2 回流至 S4。
   当前建设状态：`待规划`
5. `质量监控`
   说明：承接进度与质量监控类治理运营视图。
   当前建设状态：`待规划`

#### 4.4.3 当前判断

- 已确认该模块不仅是数据存放，还需要承担抽查与问题回流治理。
- G1 通过后的样本自动进入高质量数据池，不再额外拆出独立“正式训练库”概念。
- 高质量数据池采用“列表 + 详情 + 导出”方式管理。
- 首期“入库”与“高质量数据池”按同一治理存储表达；“导出”由数据治理侧手动发起。
- 抽查与复查任务统一由运营侧任务体系承接；其中复查任务来源于 `P5` 触发的强制复判，不单列独立模块能力。
- 抽查任务由质检员/领域工程师发起，按周/月两种周期执行。
- G2 抽查/复查判断与 R2 问题回流标记按现行知识库/规则口径执行，不单独形成脱离知识库的运营判断标准。
- 数据治理运营负责基于受影响规则/隐患类别生成强制复判候选集，不采用人工逐条挑选作为默认方式。
- 强制复判候选集默认按“受影响规则/隐患类别 + 已通过质检/已入池状态”反查历史样本；批次、时间范围只作为缩小范围的附加条件。
- 强制复判候选集形成后，由领域工程师（安全专家）确认影响范围，再正式转为复查任务。
- 首期关注上传量、待标注量、标注完成率、质检通过率、驳回率、抽查问题率、入库量。
- 数据池页面、抽查任务页面、质量监控看板分开建设。

### 4.6 知识库与规则治理

#### 4.6.1 模块定位

- 对应流程：P1、P2、P3、P4、P5。
- 模块职责：作为 PC 管理后台模块，承接标注过程中发现未覆盖隐患后的提报、审核、知识库更新与口径回写。
- 主要角色：标注员、质检员、领域工程师（安全专家）。
- 系统自动行为：口径回写通知、规则生效发布。

#### 4.6.2 主要子功能点

1. `新隐患发现登记`
   说明：在标注过程中登记未覆盖隐患，作为后续提报入口。
   当前建设状态：`待规划`
2. `新隐患提报`
   说明：由标注员/质检员提交新隐患申请。
   当前建设状态：`待规划`
3. `审核台`
   说明：由领域工程师审核新隐患申请。
   当前建设状态：`待规划`
4. `知识库维护`
   说明：维护隐患库、法规库、映射表、案例库等结构化内容，并承接首期冷启动导入。
   当前建设状态：`待规划`
5. `口径版本发布/生效`
   说明：将知识库更新结果发布为标注/质检/抽查口径版本，并负责版本生效控制。
   当前建设状态：`待规划`
6. `口径版本生效策略`
   说明：定义批次冻结、增量生效和强制复判例外规则，确保口径更新后数据有效性可判断。
   当前建设状态：`待规划`

#### 4.6.3 当前判断

- 已确认知识库与规则治理是独立并行链路，不能只作为后台附属配置处理。
- 当前阶段即建设独立知识库管理界面，因为标注能力依赖知识库口径。
- 除标注、质检外，抽查判定和问题回流标记也直接依赖现行知识库/规则口径。
- 首期知识库维护除日常维护外，还需承接已有 Excel/模板资产的冷启动导入。
- 冷启动内容默认来自现有线下隐患库、法规库和映射表，不要求平台首期承担线下内容整理。
- 口径版本采用“批次冻结、增量生效”策略：进行中的标注/质检批次沿用启动时口径，新发起批次使用最新已发布口径。
- 已完成并通过质检的数据默认继续有效；只有知识库维护时被明确标记为“强制复判”的更新，才触发存量数据复查或抽查任务。
- 强制复判是 P5 口径版本发布/生效时的决策项，不是 G1/G2 的常规入口。
- 强制复判由领域工程师（安全专家）在口径发布时决定是否触发并定义影响范围，只作用于已通过质检或已入池的历史数据，不覆盖进行中的批次。
- 强制复判先识别本次口径更新影响的规则项、隐患类别或映射关系，再按这些影响维度反查历史样本，形成复判候选集。
- 复判候选集默认只包含历史上已通过质检或已进入高质量数据池的数据；进行中的批次、未完成质检的数据不纳入候选集。
- 首期不默认全量扫描所有历史样本，也不以人工上传名单作为主路径；批次、时间范围仅作为缩小范围的附加筛选条件。
- `P5` 负责决定是否触发强制复判并发布生效口径版本；复查任务的生成与执行承接落点在数据治理运营模块。
- 复查不通过的异常样本按 R2 回流至 S4 重新标注。
- 页面结构按提报页、审核页、知识库维护页、口径发布页拆分。
- 首期纳入隐患库、法规库、隐患-法规映射表；案例库放到后续阶段。

## 5. 已确认的规划口径与后续 PRD 拆分建议

### 5.1 已确认的规划口径

- 建设范围：当前阶段建设数据接入管理、标注与质检协同、数据治理运营、知识库与规则治理；现场巡检平台作为后续来源入口保留；AI 训练与部署暂不建设。
- 模块边界：当前调整为 6 个模块，其中现场巡检平台独立为移动端模块，其他模块均为 PC 管理后台模块。
- 角色进入形态：采用“左侧菜单 + 首页工作台”模式，不同角色看到不同菜单和待办入口。
- 横向基础能力：首页工作台、待办汇聚、角色菜单/权限路由、角色权限与配置治理、全链路样本追溯作为跨模块入口能力存在，不单列为第 7 个业务模块。
- 角色治理：角色清单、角色边界、菜单权限和系统自动规则配置统一沉淀到 `数据治理平台角色与权限说明.md`，不在各 PRD 中各自定义。
- 页面策略：核心作业页面按角色拆分，共用查询/列表页可复用，作业页不共用。
- 留痕粒度：关键动作全部留痕，至少记录操作人、操作时间、动作类型、对象 ID、前后状态、备注。

### 5.2 PRD 编写顺序建议

按对后续页面设计和业务闭环的支撑优先级，建议优先编写：

1. `PRD-知识库与规则治理-知识库维护.md`
   说明：需覆盖首期 Excel/模板冷启动导入、后续维护和口径生效边界。
2. `PRD-数据接入管理-外部来源接入与PC导入.md`
3. `PRD-标注与质检协同-标注工作台.md`
4. `PRD-标注与质检协同-质检裁决与回流.md`
5. `PRD-数据治理运营-高质量数据池与导出.md`
6. `PRD-数据治理运营-抽查任务与问题回流.md`
7. `PRD-平台基础能力-角色权限与配置治理.md`
8. `PRD-平台基础能力-全链路样本追溯.md`
9. `PRD-平台门户-角色首页工作台与待办入口.md`

顺序依据：先收敛核心业务对象、状态和流转，再补横向基础能力，避免角色权限和追溯字段在业务未定前提前冻结。

现场巡检平台相关 PRD 调整为后续阶段再编写。

### 5.3 PRD 编写约束

- 每篇 PRD 必须内置“必读清单”，不要求 AI 或开发通读全部 PRD。
- 每篇 PRD 必须明确“本 PRD 不包含什么”，避免跨模块扩写。
- 若某模块下累计出现多篇 PRD 且背景说明明显变重，再考虑从本文拆出独立模块说明文档。
