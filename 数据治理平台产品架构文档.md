## 1. 文档定位与分工

- 文档目标：在业务流程基础上，沉淀平台模块边界、主要子功能点、实现要点与后续 PRD 拆分入口，作为页面设计与后续研发输入前的中间层。
- 文档边界：仅描述产品架构层信息，不展开页面交互、字段结构、接口参数、数据库结构与研发实施方案。

## 2. 架构拆分原则

### 2.1 拆分依据

- 一级模块边界以 [数据治理平台业务流程总览表.md](./数据治理平台业务流程总览表.md) §2.7“架构拆分输入清单”为准。
- 子功能点统一按“任务/页面型”拆解，围绕角色、工作台、处理对象、状态流转展开，而不是仅按流程步骤或技术能力命名。
- 所有流程引用统一使用 S/G/R/P 编号，不使用 Mermaid 变量名。

### 2.2 状态口径

- `待确认`：现有文档无法判断真实建设现状。
- `待规划`：已确认需要建设，但未见具体方案或落地证据。
- `设计中`：已有方案设计，但未见落地证据。
- `开发中`：已确认进入开发或实施阶段。
- `已上线`：已确认投入实际使用。
- `待优化`：已具备基础能力，但存在明确优化方向。
- `暂不建设`：当前阶段不纳入建设范围，仅保留流程占位。

> 根据当前已确认口径，整个平台仍处于规划阶段；如未出现更细粒度证据，模块与子功能状态默认使用 `待规划`。

### 2.3 编写约束

- 本文只记录可从现有文档推导出的模块目标、子功能点候选与状态证据。
- 不能因为某专题设计文档已存在，就直接推断平台中对应功能已上线。
- 模块状态与子功能状态在当前阶段统一以 `待规划` 为默认值；若后续出现更细粒度证据，再分别调整为 `设计中`、`开发中`、`已上线` 或 `待优化`。
- `3.x` 模块总览只保留摘要入口，不重复展开子功能点说明。
- `4.x` 各模块说明作为模块职责、子功能点、功能描述与实现要点的唯一详细正文。

## 3. 模块总览

### 3.1 模块层级关系图

```text

           |
           v
+----------------------+
|      消费输出层      |

|         S8           |
+----------------------+
```

- 主链路：数据接入管理 -> 标注与质检协同（PC管理后台） -> 数据治理运营（PC管理后台） -> AI 训练与部署。
- 侧向支撑：知识库与规则治理与标注、质检、数据治理运营双向联动。
- 双向关系含义：
  - `知识库 -> 数据治理运营`：知识库为 `G2/R2` 和强制复判提供判断依据与口径范围。
  - `数据治理运营 -> 知识库`：运营侧发现规则漂移、系统性问题或复查异常时，可反向触发 `P1-P5`。
- 终端边界：当前来源侧以巡查现场形成的图片样本提交为主，平台处理模块统一由 PC 管理后台承接；`S2` 作为扩展来源接入方向保留。
- 使用方式：该图只表达模块层级示意；详细业务流转仍以 [数据治理平台-Mermaid.mmd](./数据治理平台-Mermaid.mmd) 图源文件和 [数据治理平台业务流程总览表.md](./数据治理平台业务流程总览表.md) §2 为准。

## 4. 各模块说明

### 4.1 数据接入管理

#### 4.1.1 模块定位

- 对应流程：S1、S2、S3。
- 模块职责：承接来源样本提交、扩展来源接入、平台汇聚与样本池管理。
- 主要角色：巡查人员、数据管理员；`S2` 启用后纳入政府侧/其他业务平台等扩展来源相关方。
- 相关系统自动行为详见 [数据治理平台角色与权限说明.md](./数据治理平台角色与权限说明.md) §5。

#### 4.1.2 主要子功能点

1. `来源样本导入`
   功能描述：上传巡查图片或接收外部来源样本，完成基础校验后生成导入记录并进入后续处理流程。
   当前建设状态：`待规划`
2. `导入批次管理`
   功能描述：把导入图片关联到对应导入批次，按批次查看样本数量、处理状态和流转记录。
   当前建设状态：`待规划`
3. `来源标记管理`
   功能描述：给每个样本写入来源、批次、导入方式、提交人等字段，供后续筛选和追溯使用。
   当前建设状态：`待规划`

#### 4.1.3 实现要点与注意事项

- `S1` 是来源样本提交，线下拍照属于业务前置动作，不作为平台功能点实现。
- 批次按单次导入任务自动生成，批次边界不按时间窗或人工重划。
- 单个样本只属于一个批次，不做跨批次自动合并。
- 来源标记管理只负责写入来源锚点；跨模块的查询、关联和展示由横向基础能力中的全链路样本追溯承接。

#### 4.1.4 后续 PRD 拆分建议

- `PRD-数据接入管理-隐患图片导入.md`
- `PRD-数据接入管理-来源接入与平台汇聚.md`
- `PRD-数据接入管理-样本池与样本管理.md`

### 4.2 标注与质检协同

#### 4.2.1 模块定位

- 对应流程：S4、S5、G1、R1。
- 模块职责：承接多人多轮标注、全量质检裁决，以及不通过样本回流重标的闭环。
- 主要角色：巡查人员（当前）、质检员、领域工程师（安全专家）、数据管理员；后续可独立出数据标注团队。
- 相关系统自动行为详见 [数据治理平台角色与权限说明.md](./数据治理平台角色与权限说明.md) §5。

#### 4.2.2 主要子功能点

1. `标注池与轮次分配`
   功能描述：把新进入的样本放入标注池，并按配置轮次自动分配给不同标注人员。
   当前建设状态：`待规划`
2. `标注工作台`
   功能描述：在标注工作台查看图片，使用矩形框或多边形标注隐患区域，选择类型/等级/属性，并提交有隐患或无隐患结果。
   当前建设状态：`待规划`
3. `负样本确认`
   功能描述：当所有配置轮次都判为无隐患时，发起负样本确认；只要任一轮判为有隐患，就转入冲突裁决。
   当前建设状态：`待规划`
4. `质检裁决`
   功能描述：查看每张图片的标注结果并给出通过、驳回回流或挂起待确认结论，同时处理多轮结果冲突。
   当前建设状态：`待规划`
5. `驳回回流`
   功能描述：把质检不通过的样本退回标注环节，重新进入 S4 再标注。
   当前建设状态：`待规划`

#### 4.2.3 实现要点与注意事项

- 该模块必须形成“标注 -> 质检 -> 驳回回流 -> 再标注”的闭环。
- 标注时使用的隐患类型、等级、属性等选择项来源于 P5 口径版本发布/生效后的现行口径，不由标注人自由定义。
- 标注工作台需支持矩形框和多边形两种标注方式，具体交互细节下沉到对应 PRD。
- 样本自动进入标注池，不做人工派发；标注轮次支持最少轮次配置，并按轮次自动分配给不同人员。
- 负样本确认只在所有配置轮次都判为“无隐患”时触发；任一轮出现“有隐患”则转入冲突裁决。
- 冲突裁决归入质检裁决，由质检工作台统一处理，不单列独立功能点。
- 挂起待确认由质检员发起，领域工程师完成最终判定；因规则缺失无法裁决时转入 P1-P5。
- 标注工作台与质检工作台拆成两套独立页面。

- `PRD-标注与质检协同-标注池与轮次分配.md`
- `PRD-标注与质检协同-标注工作台.md`
- `PRD-标注与质检协同-质检裁决与回流.md`

### 4.3 数据治理运营

#### 4.3.1 模块定位

- 对应流程：S6、S7、G2、R2。
- 模块职责：作为 PC 管理后台模块，承接高质量数据沉淀、入库、导出、抽查及问题回流，形成治理运营闭环。
- 主要角色：数据治理团队、质检员、领域工程师（安全专家）、AI 工程师、数据管理员。
- 相关系统自动行为详见 [数据治理平台角色与权限说明.md](./数据治理平台角色与权限说明.md) §5。

#### 4.3.2 主要子功能点

1. `高质量数据池`
   功能描述：将 G1 通过样本自动入池，形成可筛选、可导出的高质量数据集合。
   当前建设状态：`待规划`
2. `训练数据导出`
   功能描述：查看高质量数据池样本，按条件筛选并手动导出训练候选数据。
   当前建设状态：`待规划`
3. `抽查与复查任务管理`
   功能描述：创建周/月抽查任务和强制复判复查任务，记录执行过程和结论。
   当前建设状态：`待规划`
4. `问题回流`
   功能描述：把抽查或复查发现的问题样本退回 S4，重新进入标注处理。
   当前建设状态：`待规划`
5. `质量监控`
   功能描述：展示上传量、待标注量、通过率、驳回率、抽查问题率等运营指标和任务进度。
   当前建设状态：`待规划`

#### 4.3.3 实现要点与注意事项

- G1 通过后的样本自动进入高质量数据池，不再额外拆出独立“正式训练库”概念。
- 训练数据导出只负责从高质量数据池筛选和导出，不再单独定义“入库”能力。
- 抽查与复查任务统一由运营侧任务体系承接；其中复查任务来源于 `P5` 触发的强制复判。
- G2 抽查/复查判断与 R2 问题回流标记按现行知识库/规则口径执行，不单独形成运营侧自定义标准。
- 强制复判候选集默认按“受影响规则/隐患类别 + 已通过质检/已入池状态”反查历史样本；批次、时间范围只作为附加筛选条件。

#### 4.3.4 后续 PRD 拆分建议

- `PRD-数据治理运营-高质量数据池与训练数据导出.md`
- `PRD-数据治理运营-抽查任务与问题回流.md`

### 4.4 AI 训练与部署

#### 4.4.1 模块定位

- 对应流程：S8。
- 模块职责：作为后续 PC 管理后台模块，承接高质量数据向 AI 训练与部署环节的交付，并形成与数据治理侧的衔接边界。
- 主要角色：AI 工程师、数据治理团队。
- 相关系统自动行为详见 [数据治理平台角色与权限说明.md](./数据治理平台角色与权限说明.md) §5。

#### 4.4.2 主要子功能点

1. `训练数据交付`
   功能描述：选择高质量数据并输出到训练侧，形成可用于模型训练的数据交付。
   当前建设状态：`暂不建设`
2. `训练/部署入口`
   功能描述：提供训练任务、模型部署和结果查看的统一入口。
   当前建设状态：`暂不建设`
3. `训练问题反馈登记`
   功能描述：记录训练侧反馈的数据问题，并把问题回传给数据治理侧处理。
   当前建设状态：`暂不建设`

#### 4.4.3 实现要点与注意事项

- 当前阶段不纳入建设范围，仅保留训练数据交付、训练入口和问题反馈的能力占位。
- 进入建设前，需要先明确训练侧的接收方式、训练任务管理边界和反馈回传路径。

#### 4.4.4 后续 PRD 拆分建议

- 当前阶段不编写 PRD。

### 4.5 知识库与规则治理

#### 4.5.1 模块定位

- 对应流程：P1、P2、P3、P4、P5。
- 模块职责：作为 PC 管理后台模块，承接标注过程中发现未覆盖隐患后的提报、审核、知识库更新与口径回写。
- 主要角色：标注员、质检员、领域工程师（安全专家）。
- 相关系统自动行为详见 [数据治理平台角色与权限说明.md](./数据治理平台角色与权限说明.md) §5。

#### 4.5.2 主要子功能点

1. `新隐患提报`
   功能描述：提交现有口径未覆盖的隐患，填写隐患描述、示例图片、法规依据和建议口径。
   当前建设状态：`待规划`
2. `审核台`
   功能描述：审核新隐患申请，给出通过、驳回或补充修改结论。
   当前建设状态：`待规划`
3. `知识库维护`
   功能描述：维护隐患库、法规库、映射表、案例库等内容，支持新增、编辑、删除和冷启动导入。
   当前建设状态：`待规划`
4. `口径版本发布与生效控制`
   功能描述：发布新的标注、质检和抽查口径版本，并配置版本生效和例外控制规则。
   当前建设状态：`待规划`

#### 4.5.3 实现要点与注意事项

- 知识库与规则治理是独立并行链路，不能只作为后台附属配置处理。
- 标注、质检、抽查和问题回流都依赖现行知识库/规则口径，知识库维护与口径发布必须独立建设。
- 知识库维护除日常维护外，还需支持已有 Excel/模板资产的冷启动导入。
- 口径版本采用“批次冻结、增量生效”策略：进行中的标注/质检批次沿用启动时口径，新发起批次使用最新已发布口径。
- 强制复判是 P5 口径版本发布/生效时的决策项，不是 G1/G2 的常规入口。
- 强制复判只作用于已通过质检或已入池的历史数据；复查任务的生成与执行承接落点在数据治理运营模块。

### 4.6 横向基础能力

#### 4.6.1 模块定位

- 对应范围：跨 `S1-S7`、`P1-P5` 的统一入口与治理能力。
- 模块职责：承接首页工作台、待办汇聚、角色菜单/权限路由、角色权限与配置治理、全链路样本追溯。
- 主要角色：平台内各角色。
- 相关系统自动行为详见 [数据治理平台角色与权限说明.md](./数据治理平台角色与权限说明.md) §5。

#### 4.6.2 主要子功能点

1. `首页工作台与待办入口`
   功能描述：给不同角色展示首页工作台、待办数量和常用入口。
   当前建设状态：`待规划`
2. `角色菜单/权限路由`
   功能描述：根据角色控制菜单显示和页面访问范围，并支持按角色调整。
   当前建设状态：`待规划`
3. `角色权限与配置治理`
   功能描述：配置角色、菜单权限和系统自动规则开关。
   当前建设状态：`待规划`
4. `全链路样本追溯`
   功能描述：按样本编号查看其从导入、标注、质检到入池、抽查的完整流转记录。
   当前建设状态：`待规划`

#### 4.6.3 实现要点与注意事项

- 横向基础能力不是独立业务模块，而是服务于各 PC 管理后台模块的统一支撑层。
- 角色定义、菜单权限、系统自动规则配置和全链路样本追溯统一在本层承接，不在各业务模块或 PRD 中重复定义。
- 该层允许按角色调整默认授权，但权限配置变化不改变 `S/G/R/P` 流程定义。

- 若某模块下累计出现多篇 PRD 且背景说明明显变重，再考虑从本文拆出独立模块说明文档。
