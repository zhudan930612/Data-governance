## 1. 文档定位与分工

- 文档目标：在业务流程基础上，沉淀平台模块边界、主要子功能点、当前建设状态与后续 PRD 拆分入口，作为页面设计与后续研发输入前的中间层。
- 文档边界：仅描述产品架构层信息，不展开页面交互、字段结构、接口参数、数据库结构与研发实施方案。
- 与现有文档分工：
  - 产品上下文回答“为什么做、边界是什么”
  - 业务流程回答“业务怎么走、关口和回流是什么”
  - 本文回答“平台由哪些模块承接、主要子功能点是什么、当前做到哪一步”
  - 后续 PRD 回答“这一组页面簇/工作流单元具体要做什么”

## 2. 架构拆分原则

### 2.1 拆分依据

- 一级模块边界以 `数据治理平台业务流程总览表.md` 第 2.7 节“架构拆分输入清单”为准。
- 子功能点统一按“任务/页面型”拆解，围绕角色、工作台、处理对象、状态流转展开，而不是仅按流程步骤或技术能力命名。
- 所有流程引用统一使用 S/G/R/P 编号，不使用 Mermaid 变量名。

### 2.2 状态口径

- `待确认`：现有文档无法判断真实建设现状。
- `待规划`：已确认需要建设，但未见具体方案或落地证据。
- `设计中`：已有方案设计，但未见落地证据。
- `开发中`：已确认进入开发或实施阶段。
- `已上线`：已确认投入实际使用。
- `待优化`：已具备基础能力，但存在明确优化方向。

> 根据当前已确认口径，整个平台仍处于规划阶段；如未出现更细粒度证据，模块与子功能状态默认使用 `待规划`。

### 2.3 编写约束

- 本文只记录可从现有文档推导出的模块目标、子功能点候选与状态证据。
- 不能因为某专题设计文档已存在，就直接推断平台中对应功能已上线。
- 模块状态与子功能状态在当前阶段统一以 `待规划` 为默认值；若后续出现更细粒度证据，再分别调整为 `设计中`、`开发中`、`已上线` 或 `待优化`。

+----------------------+
           |

           |
           v

- 主链路：数据来源 -> 数据接入管理 -> 标注与质检协同 -> 数据治理运营 -> AI 训练与部署。

- 使用方式：该图只表达模块层级关系；详细业务流转仍以 `数据治理平台-Mermaid.mmd` 和 `数据治理平台业务流程总览表.md` 为准。

## 4. 各模块说明

### 4.1 数据接入管理

#### 4.1.1 模块定位

- 对应流程：S1、S2、S3。
- 模块职责：承接巡检员主来源与政府端/外部补充来源的样本进入平台，并形成可处理样本池。
- 主要角色：巡检员、政府端/外部数据提供方、平台、数据管理员。

#### 4.1.2 主要子功能点

1. `来源接入管理`
   说明：承接主来源与补充来源的接入方式与接入边界。
   当前建设状态：`待规划`
2. `照片上传`
   说明：将来自不同来源的样本汇聚到平台处理链路。
   当前建设状态：`待规划`
3. `样本池管理`
   说明：形成进入标注前的可处理样本池。
   当前建设状态：`待规划`
4. `样本来源追溯`
   说明：支撑关键操作可追溯，识别样本来源与处理责任。
   当前建设状态：`待规划`

#### 4.1.3 当前判断

- 已确认的是该模块必须承接“来源接入、上传汇聚、样本池管理”这类能力。
- 未确认的是当前产品是否已存在独立的接入配置页、上传入口页、样本池页，以及具体状态流转规则。

#### 4.1.4 后续 PRD 拆分建议

- `PRD-数据接入管理-来源接入与上传.md`
- `PRD-数据接入管理-样本池与样本管理.md`

### 4.2 标注与质检协同

#### 4.2.1 模块定位

- 对应流程：S4、S5、G1、R1。
- 模块职责：承接多人多轮标注、全量质检裁决，以及不通过样本回流重标的闭环。
- 主要角色：数据标注团队、质检员/领域工程师、平台、数据管理员。

#### 4.2.2 主要子功能点

1. `标注任务分配`
   说明：将待处理样本分配给标注团队，形成任务承接关系。
   当前建设状态：`待规划`
2. `标注工作台`
   说明：承接多人多轮标注，以及有隐患/无隐患两类处理。
   当前建设状态：`待规划`
3. `负样本确认`
   说明：对无隐患样本进行确认，纳入标注工作台口径。
   当前建设状态：`待规划`
4. `质检裁决`
   说明：对标注结果进行全量质检并形成通过/驳回结论。
   当前建设状态：`待规划`
5. `驳回回流`
   说明：将质检不通过样本按 R1 回流到 S4 重新标注。
   当前建设状态：`待规划`

#### 4.2.3 当前判断

- 已确认该模块必须形成“标注 -> 质检 -> 驳回回流 -> 再标注”的闭环。
- 已确认数据管理员承担任务分配职责，但未确认分配方式、领取方式、轮次差异与留痕粒度。
- 未确认当前产品是否将标注工作台与质检工作台拆成独立页面簇。

#### 4.2.4 后续 PRD 拆分建议

- `PRD-标注与质检协同-标注任务分配与领取.md`
- `PRD-标注与质检协同-标注工作台.md`
- `PRD-标注与质检协同-质检裁决与回流.md`

### 4.3 数据治理运营

#### 4.3.1 模块定位

- 对应流程：S6、S7、G2、R2。
- 模块职责：承接高质量数据沉淀、入库、导出、抽查及问题回流，形成治理运营闭环。
- 主要角色：平台、数据治理团队、质检员/领域工程师、AI 工程师、数据管理员。

#### 4.3.2 主要子功能点

1. `高质量数据池`
   说明：沉淀经过质检通过的数据，作为治理与训练候选基础。
   当前建设状态：`待规划`
2. `入库与导出`
   说明：承接高质量数据入库及训练数据导出。
   当前建设状态：`待规划`
3. `抽查任务管理`
   说明：承接周/月抽查的发起、执行与结论记录。
   当前建设状态：`待规划`
4. `问题回流`
   说明：将抽查发现的问题样本按 R2 回流至 S4。
   当前建设状态：`待规划`
5. `质量监控`
   说明：承接进度与质量监控类治理运营视图。
   当前建设状态：`待规划`

#### 4.3.3 当前判断

- 已确认该模块不仅是数据存放，还需要承担抽查与问题回流治理。
- 已确认存在“数据导出”和“进度与质量监控”职责，但未确认其具体页面归属与指标口径。
- 未确认当前产品是否已形成独立的数据池管理页、抽查台、质量监控看板。

#### 4.3.4 后续 PRD 拆分建议

- `PRD-数据治理运营-高质量数据池与导出.md`
- `PRD-数据治理运营-抽查任务与问题回流.md`

### 4.5 知识库与规则治理

#### 4.5.1 模块定位

- 对应流程：P1、P2、P3、P4、P5。
- 模块职责：承接标注过程中发现未覆盖隐患后的提报、审核、知识库更新与口径回写。
- 主要角色：标注员、质检员/领域工程师、领域工程师/安全专家、平台。

#### 4.5.2 主要子功能点

1. `新隐患发现登记`
   说明：在标注过程中登记未覆盖隐患，作为后续提报入口。
   当前建设状态：`待规划`
2. `新隐患提报`
   说明：由标注员/质检员提交新隐患申请。
   当前建设状态：`待规划`
3. `审核台`
   说明：由领域工程师审核新隐患申请。
   当前建设状态：`待规划`
4. `知识库维护`
   说明：维护隐患库、法规库、映射表、案例库等结构化内容。
   当前建设状态：`待规划`
5. `口径回写发布`
   说明：将知识库更新结果回写至标注/质检口径。
   当前建设状态：`待规划`

#### 4.5.3 当前判断

- 已确认知识库与规则治理是独立并行链路，不能只作为后台附属配置处理。
- 已确认 `知识库体系设计.md` 已沉淀隐患库、法规库、映射表、案例库结构，但这只能证明设计已形成，不能证明产品界面已存在。
- 未确认当前是否有独立提报页、审核页、库表维护页和口径发布页。

#### 4.5.4 后续 PRD 拆分建议

- `PRD-知识库与规则治理-新隐患提报与审核.md`

## 5. 待确认项与后续 PRD 拆分建议

### 5.1 全局待确认项

- 当前已确认整个平台处于规划阶段，但各模块内部的优先级、先后顺序与拆分边界仍待明确。
- 角色进入各模块的入口形态尚未确认，例如是否按角色首页、工作台、菜单树或任务池进入。
- 多角色共用页面还是拆分独立工作台尚未确认。
- 关键状态流转与操作留痕粒度尚未确认。

### 5.2 PRD 编写顺序建议

按对后续页面设计和业务闭环的支撑优先级，建议优先编写：

1. `PRD-标注与质检协同-标注工作台.md`
2. `PRD-标注与质检协同-质检裁决与回流.md`
3. `PRD-数据接入管理-来源接入与上传.md`
4. `PRD-数据治理运营-高质量数据池与导出.md`
5. `PRD-知识库与规则治理-新隐患提报与审核.md`

### 5.3 PRD 编写约束

- 每篇 PRD 必须内置“必读清单”，不要求 AI 或开发通读全部 PRD。
- 每篇 PRD 必须明确“本 PRD 不包含什么”，避免跨模块扩写。
- 若某模块下累计出现多篇 PRD 且背景说明明显变重，再考虑从本文拆出独立模块说明文档。
