# 产品文档工作流说明

## 1. 文档定位

- 目标：沉淀一套可复用到不同产品的轻量产品文档工作流，统一“从业务基线到具体 PRD”的文档分层。
- 适用范围：适用于仍处于业务梳理、流程拆解、模块沉淀、PRD 编写阶段的产品，不覆盖页面/交互设计稿与开发实施说明。
- 当前落地对象：隐患数据治理平台。

## 2. 标准层级

| 层级 | 文档类型 | 主要回答的问题 | 直接输入 | 直接输出/下游 | 不包含 |
|---|---|---|---|---|---|
| L1 | 产品上下文 | 为什么做、业务边界是什么、角色与术语如何统一 | 业务材料、会议纪要、已有背景文档 | 业务流程、产品架构 | 具体流程步骤、页面方案、接口与库表 |
| L2 | 业务流程 | 业务怎么走、关口与回流是什么 | 产品上下文 | 产品架构、PRD 的流程边界 | 页面方案、交互细节、技术实现 |
| L3 | 产品架构 | 平台由哪些模块承接、模块做到哪一步、后续 PRD 怎么拆 | 产品上下文、业务流程、现有专题设计 | PRD 切分边界、模块现状基线 | 页面交互、字段细节、接口协议 |
| L4 | PRD | 某一组页面簇/工作流单元具体要做什么 | 产品架构、相关专题设计 | 页面设计、研发实现输入 | 全量业务背景复述、跨模块泛化设计 |

## 3. 层级分工规则

### 3.1 产品上下文

- 负责沉淀稳定业务背景、角色边界、术语口径、设计原则与外部约束。
- 只回答“业务基线是什么”，不承担流程表达和页面方案设计。

### 3.2 业务流程

- 负责沉淀流程真源及其文字解释。
- 重点说明主流程、关口、回流、并行链路与角色责任。
- 只回答“业务如何流转”，不直接映射成页面结构。

### 3.3 产品架构

- 负责承接流程并拆成产品模块、主要子功能点、状态、来源依据、待确认项。
- 子功能点统一按“任务/页面型”拆解，即围绕角色、工作台、处理对象、状态流转来拆。
- 当前阶段可兼容“模块说明”职责，不必一开始就拆成单独模块文档。

### 3.4 PRD

- 统一按“页面簇/工作流单元”拆分，不按“一个模块一个 PRD”硬拆。
- 一篇 PRD 应覆盖一组强相关页面及完整任务闭环，便于继续做页面设计和研发拆解。

## 4. 产品架构文档编写要求

### 4.1 必备字段

产品架构文档中的模块总览表至少包含以下字段：

| 字段 | 说明 |
|---|---|
| 模块名称 | 对应平台承接边界的模块名 |
| 对应流程编号 | 使用 S/G/R/P 编号，不使用 Mermaid 变量名 |
| 模块目标 | 该模块要承接的业务目标 |
| 主要子功能点 | 按任务/页面型拆解的主功能单元 |
| 当前建设状态 | 使用统一状态标签 |
| 来源依据 | 来自哪份现有文档的哪类信息 |
| 待确认项 | 当前文档无法确认、需后续补证的内容 |
| 后续 PRD 切分建议 | 后续建议拆成哪几篇 PRD |

### 4.2 状态标签

产品架构文档默认使用建设阶段型状态标签：

| 状态 | 使用口径 |
|---|---|
| 待确认 | 现有文档不足以判断真实建设现状 |
| 待规划 | 已确认需要建设，但未见具体设计或落地证据 |
| 设计中 | 已有明确方案设计，但未见落地证据 |
| 开发中 | 已确认进入开发或实施阶段 |
| 已上线 | 已确认投入实际使用 |
| 待优化 | 已具备基础能力，但有明确优化需求 |

> 原则：不能因为“文档已存在”就推断“产品已上线”。

## 5. PRD 拆分与编写要求

### 5.1 拆分原则

- 以“一个可独立设计和交付的页面簇/工作流单元”为拆分粒度。
- 优先把强耦合页面写在同一篇 PRD，避免一篇 PRD 同时覆盖多个弱相关工作台。
- PRD 命名可采用：`PRD-模块名-工作流单元名.md`。

### 5.2 PRD 开头必读清单

每篇 PRD 开头必须列出“必读清单”，至少包含：

- 当前产品的产品上下文文档
- 当前产品的产品架构文档
- 当前 PRD 直接依赖的流程/专题文档
- 本 PRD 明确不包含的相邻模块或相邻工作流

### 5.3 PRD 标准结构

每篇 PRD 至少包含以下章节：

1. 文档目标
2. 必读清单
3. 适用角色
4. 业务对象
5. 触发入口与适用场景
6. 页面簇/工作流范围
7. 核心功能需求
8. 状态流转与异常处理
9. 边界与不包含项
10. 依赖关系

## 6. AI/开发阅读路由

### 6.1 默认读取顺序

后续让 AI 或开发进入某项具体功能时，默认只按以下顺序读取：

1. 产品上下文
2. 产品架构文档
3. 当前 PRD
4. 当前 PRD 指向的专题文档

### 6.2 禁止事项

- 不要求 AI/开发先通读全部 PRD。
- 不要求 AI/开发自行猜测应该阅读哪些文档。
- 不在 PRD 中重复整份产品背景与全量模块说明。

## 7. 当前项目落地说明

隐患数据治理平台当前采用本工作流的映射关系如下：

| 层级 | 当前文档 |
|---|---|
| 产品上下文 | `数据治理平台产品上下文.md` |
| 业务流程 | `数据治理平台-Mermaid.mmd` + `数据治理平台业务流程总览表.md` |
| 产品架构 | `数据治理平台产品架构文档.md` |
| PRD | 后续按页面簇/工作流单元逐篇新增 |

## 8. 维护规则

- 新增产品时，优先复用本工作流的 4 层结构，再按产品特点微调命名。
- 某个模块下出现多篇 PRD 且背景说明明显变重时，可再拆出独立模块说明文档。
- 若某产品已经进入页面设计或研发实施阶段，也不回填“页面/交互设计稿”和“开发实施说明”到本工作流标准层中，避免层级膨胀。
