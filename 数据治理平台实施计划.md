## 实施路线图

### 第1阶段：基础建设（1-2周）

**目标**：建立"最小可用数据底座"，为后续开发奠定基础

#### 1.1 字段与分类分级冻结（优先级：Critical）

**负责人**：产品经理 + 领域专家 + 领域工程师

**目标**：先冻结数据结构，避免后续录入返工。

**交付物**：
- [ ] 《标准库+隐患库字段字典v1.0》
  - 标准库字段（标准名、编号、版本、生效状态、章、条）
  - 隐患库字段（隐患ID、描述、风险等级、整改建议、关键词、位置描述）
  - 映射字段（隐患条目 <-> 条款级引用）
- [ ] 《分类分级规则v1.0》
  - 三级结构：大类 -> 子类 -> 隐患条目
  - 风险等级与整改时限定义
  - v1.0首批优先域：触电、高坠、物体打击、机械伤害、动火/消防

**验收标准**：
- 字段冻结评审通过（产品、领域、技术三方签字）
- 分类分级无冲突（抽样30条条目映射一致性=100%）

**依赖**：无

#### 1.2 首批标准录入与审核机制建立（优先级：Critical）

**负责人**：领域工程师（3-5年）+ 领域审核专家（5-10年）

**目标**：建立可用的首批标准库和隐患库条目，形成"录入-审核"闭环。

**交付物**：
- [ ] 《标准库首批条目清单v1.0》（按优先域分批录入）
- [ ] 《隐患库首批条目清单v1.0》（含整改建议与风险等级）
- [ ] 《条款级引用规范v1.0》
  - 每条隐患必须绑定：标准名+编号+章+条
- [ ] 《录入与审核职责说明》
  - 3-5年人员：条目录入与初校
  - 5-10年人员：专业复核与发布审核

**验收标准**：
- 首批优先域条目录入完成率=100%
- 条款级引用必填率=100%
- 审核通过条目可直接用于标注检索与报告匹配

**依赖**：1.1字段冻结

#### 1.3 拍摄SOP制定与流程联调（优先级：Critical）

**负责人**：产品经理 + 领域专家 + 应用开发工程师

**目标**：打通"现场采集 -> 多人多轮标注 -> 全量质检 -> 入库"最小闭环。

**交付物**：
- [ ] 《拍摄SOP v1.0》
  - 每点位多角度拍摄要求
  - 必拍项与选拍项
  - 语音备注模板
  - 无隐患（负样本）上传规范
- [ ] 《质检规则v1.0》
  - 同图多人多轮标注规则
  - 全量质检与打回规则
- [ ] 联调与试跑报告（样板点）
  - 自有巡检数据链路全流程走通

**验收标准**：
- 样板点试跑完成并闭环
- 现场巡检动作平均时长满足"极简动作"目标
- 负样本与新增隐患提报链路可执行

**依赖**：1.1、1.2

#### 1.4 法规跟踪与季度评审机制（优先级：Major）

**负责人**：领域专家 + 产品经理

### 第2阶段：对接规范和联调（2-4周）

**目标**：完成外部系统数据契约，确保数据能够顺利接入

#### 2.1 政府端"日常巡查"模块对接规范（优先级：Critical）

**负责人**：应用开发工程师 + 数据工程师

**交付物**：
- [ ] 《政府端日常巡查模块对接规范v1.0》
  - API接口定义（RESTful/GraphQL）
  - 请求/响应数据格式（JSON Schema）
  - 字段映射表（政府端字段 → 数据治理平台字段）
  - 增量同步策略（基于时间戳/版本号）
  - 全量同步策略（初始化场景）
  - 失败重试机制（指数退避）
  - 幂等性设计（基于唯一键去重）
  - 数据校验规则（必填字段、格式校验）
  - 错误码定义和处理流程

**验收标准**：
- **规范完整性**：覆盖所有数据交换场景（同步、查询、回调），包含API定义、数据格式、容错机制、错误码（验证方法：场景清单逐项检查；验证人：数据工程师）
- **字段映射完整性**：政府端所有字段100%映射到数据治理平台（验证方法：字段清单逐项对比；验证人：数据工程师）
- **容错机制有效性**：模拟网络异常、数据异常等5种异常场景，系统能够正确处理并记录日志（验证方法：异常场景测试；验证人：应用开发工程师）
- **跨团队确认**：政府端团队负责人签字确认接口规范

**依赖**：政府端"日常巡查"模块开发进度

**风险**：
- 政府端模块可能尚未开发完成，接口规范可能变更
- 建议：先定义接口规范，政府端按规范开发；如政府端已有接口，则需要适配

**验收标准**：

#### 2.4 联调验证和测试（优先级：Critical）

**负责人**：应用开发工程师 + 数据工程师

**交付物**：
- [ ] 联调测试报告
  - 接口联调通过率

- **审计日志完整性**：覆盖所有关键操作（数据访问、数据修改、数据导出、权限变更），日志不可篡改（验证方法：操作清单逐项检查；验证人：数据平台与安全负责人）
- **版本冻结**：数据平台与安全负责人签字确认v1.0版本

**依赖**：无

#### 3.1.1 合规控制矩阵（优先级：Critical）

**负责人**：数据平台与安全负责人 + 产品经理

**交付物**：
- [ ] 《数据治理平台合规控制矩阵v1.0》
  - 法律法规清单
    - 《中华人民共和国数据安全法》
    - 《中华人民共和国个人信息保护法》
    - 《网络安全法》
    - 行业特定规范（如有）
  - 合规要求映射表
    - 法规条款 → 系统控制项 → 实施状态
  - 关键控制点
    - 数据采集：知情同意、最小必要原则
    - 数据存储：加密存储、访问控制
    - 数据使用：用途限制、授权审批
    - 数据传输：加密传输、传输审计
    - 数据删除：安全删除、删除审计
  - 合规检查清单
    - 上线前必检项（20+项）
    - 运行期定期检查项（10+项）
  - 合规风险评估
    - 高风险项识别
    - 风险缓解措施
    - 应急响应预案

**验收标准**：
- **法规覆盖度**：覆盖所有适用法律法规（至少3部：数据安全法、个人信息保护法、网络安全法），每部法规关键条款100%映射到系统控制项（验证方法：法规条款清单逐项对比；验证人：数据平台与安全负责人）
- **控制项可执行性**：每个控制项有明确的实施方法和验证标准（验证方法：控制项清单逐项审查；验证人：数据平台与安全负责人）
- **风险评估完整性**：识别所有高风险项（至少5项），每项有明确的缓解措施和应急预案（验证方法：风险清单逐项检查；验证人：数据平台与安全负责人）
- **跨部门确认**：法务部门/合规部门负责人签字确认

**依赖**：安全架构设计

#### 3.1.2 数据准入规范（优先级：Critical）

**负责人**：数据平台与安全负责人 + 数据工程师

**交付物**：
- [ ] 《数据准入规范v1.0》
  - 数据来源准入标准
    - 政府端数据：需签订数据共享协议，明确数据用途和保护责任
    - 第三方服务机构数据：需签订数据处理协议，确保数据采集合法性
    - 外部购买数据：需验证数据来源合法性，要求提供数据授权证明
  - 数据质量准入标准（分层策略）
    - **核心训练集**（高质量门槛）：分辨率 ≥ 720p，模糊度评分 ≥ 60，曝光度评分 ≥ 60
    - **泛化补充集**（放宽门槛）：分辨率 ≥ 480p，模糊度评分 ≥ 40，曝光度评分 ≥ 40；标记"泛化补充"域属性，训练时单独处理以提升模型泛化能力
    - **说明**：放宽低分辨率样本的准入是为了覆盖不同设备采集场景，避免训练数据分布偏移导致泛化能力受损
    - 内容质量：必须包含施工场景，不得包含无关内容
    - 完整性：必须包含拍摄时间、拍摄地点等基础元数据
  - 数据安全准入标准
    - 不得包含个人敏感信息（人脸、车牌需脱敏处理）
    - 不得包含涉密信息（军事设施、敏感地点）
    - 传输过程必须加密
  - 准入流程
    - 数据来源审核 → 数据质量检测 → 数据安全检测 → 准入决策 → 入库
  - 拒绝准入处理
    - 拒绝原因记录
    - 数据源反馈机制
    - 整改后重新申请流程
  - 准入审计
    - 准入日志记录（数据来源、准入时间、审核人、审核结果）
    - 定期准入质量分析（按数据源统计准入率、拒绝原因分布）

**验收标准**：
- **准入标准可执行性**：3类数据来源（政府端、第三方、外部购买）准入标准明确，包含授权证明、质量标准、安全标准（验证方法：标准清单逐项审查；验证人：数据平台与安全负责人）
- **准入流程可追溯性**：每条数据准入决策100%可追溯（数据来源、审核人、审核时间、审核结果、拒绝原因）（验证方法：准入日志抽查100条；验证人：数据工程师）
- **自动化检测覆盖率**：自动化检测规则覆盖率 > 80%（技术质量、内容质量、安全检测），人工审核 < 20%（验证方法：准入流程分析；验证人：数据工程师）
- **合规通过率**：数据准入合规通过率 = 100%（验证方法：抽检50条 + 全量元数据校验；验证人：数据平台与安全负责人）
- **版本冻结**：数据平台与安全负责人签字确认v1.0版本

**依赖**：安全架构设计、合规控制矩阵

## 风险管理

### 高风险项

1. **政府端"日常巡查"模块开发延期**
   - 影响：政府端数据接入后置，影响来源扩展节奏，但不阻塞v1.0主链路
   - 缓解措施：
     - 优先使用事故预防服务数据（自建巡查队伍、第三方服务机构）完成v1.0
     - 与政府端团队保持密切沟通，及时了解开发进度
     - 政府端仅作为v1.x扩展项，按接口就绪后并行接入

2. **隐患问题库定义存在争议**
   - 影响：标注规范无法确定，标注工作无法开展
   - 缓解措施：
     - 组织多轮专家评审，尽早达成共识
     - 采用迭代方式，先冻结核心类型，后续逐步补充
     - 建立版本管理机制，允许后续调整

3. **标注质量不达标**
   - 影响：数据集质量差，模型性能不达标
   - 缓解措施：
     - 加强标注员培训
     - 提高交叉验证和质检比例
     - 引入标注员能力画像和绩效考核
     - 及时反馈和优化标注规范

### 中风险项

1. **对接接口变更**
   - 影响：已开发的对接功能需要返工
   - 缓解措施：
     - 尽早与外部系统团队确定接口规范
     - 采用适配器模式，降低接口变更影响
     - 建立接口变更通知机制

2. **安全合规验收不通过**
   - 影响：平台无法上线
   - 缓解措施：
     - 提前与安全团队沟通，明确验收标准
     - 分阶段验收，及时发现和解决问题
     - 预留充足的整改时间
