# 数据治理平台业务工作流程v1.0

> 文档说明：基于产品讨论梳理的整体业务工作流程，聚焦业务视角，不涉及技术实现细节。
> 规划版本：v1.0
> 文档版本：v1.3
> 更新日期：2026-03-19

---

## 一、业务目标

**平台目标**：构建安责险相关的数据资产治理体系，通过标准化的数据标注、清洗、验证流程，为AI隐患识别模型训练提供高质量数据集，最终实现自动化隐患识别能力。当前阶段以小散工程安全隐患为切入口，后续逐步扩展到安责险全领域。

**商业闭环**：数据治理 → AI模型训练 → 自动化隐患识别 → 提升事故预防服务质量 → 降低保险赔付风险 → 持续获得保险业务

---

## 二、数据来源与职责边界

| 数据来源 | 责任方 | 照片质量 | 可控程度 | 标注方式 |
|---------|--------|---------|---------|---------|
| 事故预防服务巡查 | 我们自己 | 可通过巡查标准规范提升 | 高（主要来源） | 专职标注员标注（巡检员初步标注仅作参考） |
| 政府端日常巡查 | 政府人员 | 参差不齐，被动接收 | 低（辅助来源） | 专职标注员标注 |
| 外部购买数据 | 外部 | 质量未知 | 低（补充来源） | 专职标注员标注 |

**核心结论**：事故预防服务数据是我们可控的主要来源，应通过建立巡查标准（隐患清单、巡查事项、拍摄规范）从源头保障照片质量。

## 三、业务工作流程

### 流程总览

```
【数据来源】
  巡检员（主来源）现场巡查·拍照·语音备注
  政府端 / 外部数据（后置）
        |
        v
  照片上传平台
        |
        v
┌─────────────────────────────────────────┐
│ 平台处理（PC端）                          │
│                                         │
│  多人多轮标注                             │
│  有隐患：框选+类型+等级                    │
│  无隐患：负样本确认                        │
│        |                                │
│        v                                │
│  全量质检（质检员裁决）                    │
│    |通过          |不通过                 │
│    v              v                     │
│  高质量数据池/入库  打回修改               │
│    |  |周/月抽查   附错误说明+参考示例 <---┤
│    |  v                ^                │
│    |  定期抽查（独立运营）                 │
│    |  |发现问题         |                │
│    |  +----------------+                │
└────|────────────────────────────────────┘
     |
     v
┌─────────────────────────────────────────┐
│ AI训练与部署                              │
│                                         │
│  数据积累达到训练门槛                      │
│        |                                │
│        v                                │
│  AI模型训练                              │
│        |                                │
│        v                                │
│    评估达标？                             │
│    |是            |否                    │
│    v              v                     │
│  部署到巡查业务平台  输出困难样本清单        │
│  +部署到PC标注平台  |（回流到标注）          │
│  作为辅助标注建议    |                     │
└───────────────────|─────────────────────┘
                    |
        +-----------+ （回到多人多轮标注）

【AI辅助标注说明】
  AI模型部署后，同步接入PC端标注平台：
  · 标注员打开图片时，AI自动给出隐患识别建议（框选位置+类型+等级）
  · 标注员可采纳（一键确认）、修改（调整框/类型/等级）或忽略（自行标注）
  · AI建议仅作参考，最终标注结果以标注员操作为准，仍需经过全量质检

【知识库回流（并行）】
  照片上传平台 → 发现未覆盖隐患 → 提报→审核 → 更新标准库/隐患库 → 影响标注
```

### 阶段一：现场采集（AI上线前）

```
【事故预防服务巡检员】
赴现场 → 按巡查标准识别风险点
    ↓
    ├── 发现隐患 → 拍照记录
    │   移动端操作（v1.0）
    │   · 拍照 + 语音输入（描述隐患类型、存在问题）
    │   · 语音自动转文字，作为后续平台标注参考备注
    │   · 不在移动端做正式框选标注
    │
    └── 未发现隐患 → 拍摄现场照片上传（作为负样本保留）
        · 照片需为有效施工场景（非模糊/曝光不足）
        · 上传时标注"本次巡查无隐患"

【政府端巡查人员】
按政府自己流程巡查拍照 → 照片通过接口同步到平台（被动接收）
```

### 阶段二：平台数据处理

```
所有来源照片汇入平台
        ↓
① 标注任务领取（早期极简模式）
   · 标注员（初期可能即为巡检员本人）直接从照片池中逐一取照片进行标注
   · 早期不设任务分配和进度管理，标注员自主领取
   · 待标注团队扩大后再引入任务管理机制
        ↓
② 多人多轮标注（平台PC端）
   · 系统配置同一张照片的标注轮次和覆盖范围
   · 多名标注员独立完成同图标注，生成多轮标注结果
   · 巡检员语音备注作为参考，不替代正式标注
   · 每轮标注结果分两类：
     - 有隐患：框选隐患区域 + 选择隐患类型 + 标注风险等级
     - 无隐患（负样本）：确认"无隐患"，不绘制边界框
        ↓
③ 质检审核（平台PC端，全量）
   · 质检员对全部样本进行审核，不采用抽检
   · 复核定义：对同图多轮标注结果进行确认与裁决
   · 通过 → 数据进入高质量数据池并入库
   · 不通过 → 打回修改，同时告知：
     - 具体错在哪里（框选不准 / 类型选错 / 漏标隐患等）
     - 附同类隐患正确标注示例作为参考
   · 质检通过结果作为训练样本入库的唯一有效口径（标准库/隐患库的类目扩充走独立的专业审核通道，见阶段四）
        ↓
④ 定期抽查（运营阶段）
   · 对已入库样本按周/月执行定期抽查
   · 抽查目标：发现系统性误标、规则漂移、条款失效导致的口径偏差
   · 抽查通过 → 维持入库状态
   · 抽查发现问题 → 回流打回，进入重新标注/复核链路
```

### 阶段三：AI训练与部署

```
高质量数据持续积累
        ↓ 达到训练门槛
AI模型训练（内部AI工程师负责）
        ↓ 评估达标
部署到巡查业务平台 + 接入PC端标注平台（辅助标注）
        ↓
· 巡检员现场使用AI辅助识别隐患
· 标注员打开图片时，AI自动给出隐患识别建议（框选位置+类型+等级）
· 标注员可采纳（一键确认）、修改（调整框/类型/等级）或忽略（自行标注）
· AI建议仅作参考，最终标注结果以标注员操作为准，仍需经过全量质检
```

### 阶段四：新增隐患回流与AI上线后的流程切换

```
【新增隐患回流（v1.0起执行）】
说明：此流程扩充的是隐患类目/标准库，不是训练样本入库，不经过质检通道。
现场发现库未覆盖隐患
        ↓
新增提报（附照片/语音/场景说明）
        ↓
专业审核（领域工程师 + 审核专家）
        ↓
更新标准库/隐患库（保留版本与状态）
        ↓
同步到标注端检索与匹配逻辑

【AI上线后的现场采集（升级版）】
巡检员赴现场 → 拍照
        ↓
AI自动预标注（识别隐患位置和类型）
        ↓
巡检员确认或修正
· 预标注正确 → 直接确认
· 预标注错误 → 手动清除，自行重新框选
        ↓
照片连同标注上传平台

【平台处理阶段】
· 流程与AI上线前一致，巡检员确认/修正结果仍作为专职标注员参考输入
· 初期保守策略：不因AI置信度高而跳过专职标注
· 待模型稳定、巡检员培训达标后，可评估是否开放高置信度样本快速通道
```

### 持续改进闭环

```
AI模型评估不达标
        ↓
输出问题类型 / 困难样本清单
        ↓
专职标注员针对性补充标注 → 质检 → 重新入数据池
        ↓
模型重新训练 → 持续迭代
```

## 四、关键角色与分工

| 角色 | 阶段 | 核心职责 |
|------|------|---------|
| 巡检员 | 现场采集 | 按巡查标准现场拍照并语音备注；不承担正式标注 |
| 专职标注员 | 平台标注 | 对所有照片进行正式标注（巡检员语音备注作为参考输入） |
| 质检员 | 平台质检 | 审核所有标注，通过或打回（附错误说明和参考示例） |
| AI工程师 | AI训练 | 使用标注数据训练和评估AI模型，反馈数据质量问题 |

> **角色演进说明**：初期巡检员与标注员可能是同一个人。典型工作模式为：白天赴现场巡查拍照上传，回到电脑后再登录平台完成正式标注。中后期随业务规模扩大，两个角色会逐步分开，由专职标注员承担平台标注工作。无论角色是否合并，标注工作均在 PC 端平台完成，质检审核机制照常执行。

## 五、关键业务决策（已确认）

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 照片质量保障方式 | 建立巡查标准（隐患清单、巡查事项、拍摄规范） | 从源头解决，比平台端被动过滤更有效 |
| 移动端采集内容 | 拍照 + 语音输入转文字（仅作备注参考） | 符合计件制场景，现场动作极简；正式标注在PC完成 |
| 谁来做平台标注 | 专职标注员对所有照片正式标注；初期可由巡检员兼任，中后期角色分开 | 保证标注专业性；初期节约人力成本，中后期再引入专职标注员 |
| 政府端数据处理 | 同一套流程，由专职标注员承担标注 | 流程统一，降低管理复杂度 |
| 谁来做质检 | 专职质检员，单独岗位 | 避免自检盲区，保证数据质量 |
| 标注与复核机制 | v1.0采用同图多人多轮标注，复核即质检员对多轮结果进行确认 | 先提高标注一致性，再由质检统一裁决 |
| 质检机制 | v1.0采用全量质检，不使用抽检 | 当前阶段以质量优先，降低错误样本入库风险 |
| 定期抽查机制 | 入库后按周/月执行运营侧定期抽查 | 持续识别系统性误差与规则漂移，防止问题样本长期滞留 |
| 质检反馈方式 | 告知具体错误原因 + 附正确标注示例 | 减少返工次数；帮助标注员能力提升 |
| 库未覆盖隐患处理 | 现场提报 -> 审核 -> 更新标准库/隐患库 | 保证知识库持续迭代，避免盲区长期存在 |
| AI上线后流程切换 | AI预标注 → 巡检员确认/修正（可手动清除重标）；同步接入PC端标注平台，AI给出辅助标注建议，标注员可采纳/修改/忽略 | 平滑过渡；保留人工兜底；提升标注效率 |
| AI上线后标注处理策略 | 初期保守：巡检员确认结果仍作参考，专职标注员正式标注 | 避免初期模型误导；待模型稳定后再评估是否开放高置信度快速通道 |
| 负样本处理 | 保留无隐患但场景有效的照片，专职标注员确认"无隐患"后进入数据池 | 负样本对控制误报率至关重要；由专职标注员确认，避免漏标隐患误入负样本池 |
| 标注方法选型 | 边界框（Bounding Box）+ 分类属性（隐患类型 + 风险等级） | 目标检测任务主流格式，与 YOLO 等模型直接匹配；相比多边形效率高 3-5 倍；无边界歧义，质检标准易量化；施工隐患识别只需定位位置和判断类型，不需要精确轮廓 |

## 六、待实施合规控制点

> 以下合规节点早期暂不执行，记录于此作为后期平台建设时的需求依据。

### 数据准入校验（照片进入平台前）

| 控制点 | 要求 | 适用来源 |
|--------|------|---------|
| 外部购买数据授权校验 | 核查授权证明和用途边界，超出授权范围不得进入训练集 | 外部购买数据 |
| 个人信息脱敏 | 含人脸、车牌等个人信息的照片需脱敏处理后方可进入标注流程 | 所有来源 |
| 接口授权确认 | 确认政府端、第三方平台的数据接口授权范围 | 政府端、第三方 |

### 数据出库审批（数据集导出前）

| 控制点 | 要求 |
|--------|------|
| 导出审批 | 数据集导出需经指定角色审批 |
| 审计日志 | 记录操作人、操作时间、导出数据范围，日志保留 ≥ 180 天 |
| 数据用途说明 | 明确本次导出数据用于哪个模型版本训练 |
